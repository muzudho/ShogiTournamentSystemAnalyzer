# プログラム4パート構成 第2ステップ実行計画

## 目的

第1ステップで入力導線を `RequestFileCheck` / `ManualInput` / `RequestFileCreate` へ寄せた。
第2ステップでは、残っている `Analysis` を `Program.cs` から分離する。

## 現状

`Program.cs` には次の責務が残っている。

- アプリケーション予算の開始
- プログラム説明の表示
- 入力導線の準備
- 分析フロー選択
- 対象ルール選択
- 分析フローの dispatch
- 分析後の要求ファイル作成

このうち「分析フロー選択」「対象ルール選択」「dispatch」は、ライフサイクル上の `Analysis` として独立させる。

## 今回やること

1. `Application/Execution/Analysis.cs` を追加する。
2. `Program.RunAnalysis()` の中身を `Analysis.Run()` へ移す。
3. `Program.cs` から `RunAnalysis()` を削除する。
4. `Program.cs` では `Analysis.Run()` を呼ぶだけにする。
5. 不要になった using を削る。

## 今回やらないこと

- `AnalysisFlowDispatcher` の中身の整理
- 各モードの入力処理変更
- `RequestFileCheck` の検査専用化
- STSAInput の仕様変更

## 完了条件

- `Program.cs` がオーケストレーション中心になる。
- `Analysis` の入口が `Application/Execution/Analysis.cs` にある。
- `dotnet build` が成功する。
- 空ルールの最小手動入力が動く。
