# プログラム4パート構成 第144ステップ実行計画

## 目的

`Application/Analysis` の過剰な包みを外し、［3大域］のアプリケーション実行層を `Application` 直下へ近づける。

現状は `Application/Analysis/Domains/...` と `Domain/...` に似た名前のフォルダーが並んでおり、`Domain` という言葉がアプリケーション層とドメイン層の両方に出てくる。実体として `Application/Analysis/Domains` に入っているのは、純粋ドメインではなく、シミュレーション域、最終順位付け域、大会品質評価域をアプリとして実行する use case / mode / mainline 群である。

第144ステップでは、まず `Application/Analysis/Domains` 配下の3大域を `Application` 直下へ昇格し、`Analysis` と `Domains` の二重ラップを減らす。

## 現状

- `Application/Analysis` 直下には `Boundaries`、`Domains`、各 dispatcher、`AnalysisExecutionContext` がある。
- `Application/Analysis/Domains` 配下には、主に次の領域がある。
  - `Simulation`
  - `FinalRanking`
  - `TournamentQualityEvaluator`
  - `TournamentFinalState`
- `Domain` 直下にも `Simulation`、`FinalRanking`、`TournamentQualityEvaluator`、`TournamentFinalState` などがあり、フォルダー名だけでは層の違いが読み取りづらい。
- `Application/Analysis/Domains/*` の namespace 参照は、`ApplicationWorkflow`、request parser、各 dispatcher、各 use case / mode / mainline の相互参照に広く出ている。

## 方針

- このステップでは、計算ロジックと実行順序を変えない。
- `Domain/...` は純粋ドメイン層として残す。
- `Application/Analysis/Domains/...` は `Application/...` へ昇格する。
- `Application/Analysis/Boundaries` と `Application/Analysis` 直下 dispatcher/context は、このステップでは原則として残す。
- `AnalysisRequestDispatcher` などのクラス名変更は、このステップでは行わない。移動だけに寄せて差分の意味を保つ。
- namespace はフォルダー移動に合わせて更新する。

## 実施内容

1. 移動対象ファイルを確認する。
   - `rg --files ShogiTournamentSystemAnalyzer/Application/Analysis/Domains`
   - 移動前に `git status --short` が空、または既存差分が今回対象外であることを確認する。
2. フォルダーを移動する。
   - `Application/Analysis/Domains/Simulation` を `Application/Simulation` へ移す。
   - `Application/Analysis/Domains/FinalRanking` を `Application/FinalRanking` へ移す。
   - `Application/Analysis/Domains/TournamentQualityEvaluator` を `Application/TournamentQualityEvaluator` へ移す。
   - `Application/Analysis/Domains/TournamentFinalState` を `Application/TournamentFinalState` へ移す。
3. namespace 宣言を更新する。
   - `Application.Analysis.Domains.Simulation` を `Application.Simulation` へ置換する。
   - `Application.Analysis.Domains.FinalRanking` を `Application.FinalRanking` へ置換する。
   - `Application.Analysis.Domains.TournamentQualityEvaluator` を `Application.TournamentQualityEvaluator` へ置換する。
   - `Application.Analysis.Domains.TournamentFinalState` を `Application.TournamentFinalState` へ置換する。
4. using を更新する。
   - dispatcher、request parser、manual reader、各 use case / mainline 内の using を新 namespace へ合わせる。
   - 可能なら未使用 using を削除する。
5. 空になった `Application/Analysis/Domains` を削除する。
   - `Application/Analysis` 自体は、このステップでは dispatcher/context と `Boundaries` の置き場として残す。
6. ドキュメントを更新する。
   - `docs/開発/README.md` にこの計画ファイルを追加する。
   - 実施後は `docs/続きはここから.md` に完了内容、検証結果、次の候補を短く追記する。

- `Application/Analysis/Boundaries` の移動。
- `AnalysisExecutionContext`、`AnalysisRequestDispatcher`、各 `*RequestDispatcher`、各 `*FlowDispatcher` の移動または改名。
- `SimulationDomain`、`FinalRankingDomain`、`QualityEvaluationDomain` などのクラス名変更。
- `Domain/...` 配下のフォルダー構成。
- STSAInput/5 の入力形式、要求ファイル形式、出力ファイル形式。
- シミュレーション、最終順位付け、品質評価の計算ロジック。
