# プログラム4パート構成 第124ステップ実行計画

## 目的

`STSAInput/5` の `RuleProfileAttributes` を、互換 `RuleProfileMode` へ戻さず parser 内部で直接持ち回る。

第123ステップで `ParseStepRequest(...)` は `RuleProfileAttributes` を受け取る形になったため、`STSAInput/5` 読み取り時に `RuleProfileMode` を挿入する正規化処理は不要になっている。

## 実施内容

1. `StsaInput4RequestParser.TryParse(...)` で、入力形式が `STSAInput/4` か `STSAInput/5` かを判定する。
2. `STSAInput/5` の単一ステップでは `RuleProfileAttributes` セクションを直接読み、`ParseStepRequest(...)` へ渡す。
3. `STSAInput/5` の複数ステップでは `*Step.RuleProfileAttributes` セクションを直接読み、各 step の `ParseStepRequest(...)` へ渡す。
4. `NormalizeAttributeFormatToCompatibilityFormat(...)` と `InsertCompatibilityRuleProfileModes(...)` を削除する。
5. `STSAInput/5` 仕様から「互換ラベルへ正規化する」制約を外す。

## 変えないこと

- `STSAInput/4` の `RuleProfileMode` 入力互換。
- `STSAInput/5` のファイル表記。
- Writer と依頼ログの既定出力形式。
- 標準 / 本戦 / 複数ステップの計算ロジック。
- 出力 CSV / Markdown の内容。

## 実施結果

- `StsaInput4RequestParser` は `STSAInput/5` の `RuleProfileAttributes` を互換 `RuleProfileMode` へ変換せず、`RuleProfileAttributes` のまま step request 判定へ渡すようになった。
- `STSAInput/5` 用の互換 `RuleProfileMode` 挿入処理を削除した。
- `STSAInput/5` 仕様の現在の制約を更新した。
