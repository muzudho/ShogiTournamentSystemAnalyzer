# ［大会ルール作成］の一般化の案

［タイム］の定義：  

```plaintext
型：　非負の整数。
スパン名：　秒、分、時間だったり、ラウンド数だったり、使い方は任意。ただのラベル文字列。

説明：
	ただのループカウンター。
	後述の TimeLoop （for文）のループカウンター。1ずつ増える。
```

［プレイヤー］の構造：  

```plaintext
データ：
	名前：　アリスさん、ボブさんなど
	レーティング：　Elo レーティングなど、数値で表されるプレイヤーの強さ
	先手勝ち数：　先手で対局に勝った回数
	後手勝ち数：　後手で対局に勝った回数
	先手負け数：　先手で対局に負けた回数。不戦敗か、棄権かはこだわらない。
	後手負け数：　後手で対局に負けた回数。不戦敗か、棄権かはこだわらない。
	先手分け数：　先手で対局が引き分けになった回数
	後手分け数：　後手で対局が引き分けになった回数
	状態：
		ペアリング待ち：　ペアリングされるのを待っている状態
		対局待ち：　ペアリング後の状態を参照
		対局中　：　ペアリング後の状態を参照
		対局終了：　ペアリング後の状態を参照
```


［ペア］の構造：  

```plaintext
データ：
	先手：　アリスさん（プレイヤー）
	後手：　ボブさん（プレイヤー）
	対局開始までの残りタイム：　１（タイム）
	対局終了までの残りタイム：　３（タイム）
	対局結果：　先手勝ち、後手勝ち、引き分け
		／／　［先手］［後手］［先手勝率］［レーティング］の情報と［乱数］があれば算出できるから、
		／／　実は［対局結果］は生成時にもう決まっている。
	進捗状態：
		対局待ち：　生成されてから、［対局開始までの残りタイム］が０でない間カウントダウン。
		対局中　：　［対局開始までの残りタイム］が０になってから、［対局終了までの残りタイム］が０でない間カウントダウン。
		対局終了：　［対局終了までの残りタイム］が０になったとき。カウントストップ。
```


［プレイング・プール］の構造：  

```
データ：
	［ペア］のリスト

メソッド：
	空か（）
	ペアリング（）	／／　新しいペアをリストに追加する。
	更新（）		／／　リストに含むすべての ［ペア］のタイムを１進める。（１カウントダウン）
	全ての対局が終了しているか（）
	対局が終了しているペアを全て削除（）
	ペアリング待ちの２名をペアリングしてリストに追加する（）
```


疑似コード：  

```plaintext
// ループの開始前のペアリング
playingPoolList = Serialize( tableText )		// 様々な書式で対局表を入力　→　ペアのリストに変換。直列化。

// ループ
TimeLoop: for (t = 0; t < ∞; t += 1) {

	playingPoolList.更新()		// タイムを１進める。対局の進捗状態も更新される。
								// タイムはラウンドかも知れないし、分かもしれないが、気にしなくていいはず。

	// ［ラウンド制］の大会は、［ペア］の［対局終了までの残りタイム］で自動的に［対局中］になるので、実は何もしなくていい。
	// 終了している［ペア］は、［終了リスト］などを作って、そこに移すなどして管理するといいかも知れない。

	// 何をするかは、ルールによって変わる。
	//
	//		- 例えば、アリーナ制では、 `playingPoolList.ペアリング待ちの２名をペアリングしてリストに追加する（）`
	//		- 例えば、スイス式では、勝ち星などを考慮して、対局待ちのプレイヤーをペアリングする。
	//		- 優勝が決まった時点で大会終了するなら、ループを抜ける処理があるかも知れない。
	onTick()

	playingPoolList.対局が終了しているペアを全て削除（）

	if (playingPoolList.空か())
	{
		break TimeLoop;		// ループを抜ける。
	}
}

おわり。
```
