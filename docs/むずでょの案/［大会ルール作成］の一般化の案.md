## 初期移行の境界案

いきなり既存の実装を全部置き換えると壊れやすいので、最初は次の境界で移行する。

### 今回の初期対象

- 上位モードは引き続き **対局シミュレーション / 品質評価** を使う
- 新しい大会進行フレームワークは、まず **対局シミュレーション** 側へ並走追加する
- 最初に移すのは、現行の **Standard 相当** の固定対局進行
- 新基盤では `PlayerEntry` / `StageEntry` / `TournamentMatchRecord` / `TournamentState` を共通モデルにする
- `STSAInput/2` は実行入口として残し、必要なら外部ルールファイルや CSV 群を参照する

### 今回はまだ移さないもの

- 品質評価モード
- 既存 FinalStage 実装の完全移行
- DSL の万能化
- 既存の `Match(first, second)` ベースの計算経路の即時削除

### 並走方針

- 既存の Standard / FinalStage は残したままにする
- 新しい大会進行フレームワーク用のシミュレーション入口を別に追加する
- 旧経路と新経路の両方をビルド・実行できる状態をしばらく維持する
- 旧 Standard と新 Standard 相当の出力を比較できるようにする

### 最初の実装順

1. 大会進行フレームワークの共通モデルを追加
2. ルール抽象（ペアリング、順位、終了条件、結果確定）を追加
3. 固定対局を動かせる最小エンジンを追加
4. Players / Stages / TournamentMatchRecords の CSV 読み込みを追加
5. 最小 DSL と、その読み込みを追加
6. 新しい対局シミュレーション入口を追加
7. 現行 Standard 相当を新基盤へ載せる
8. 旧経路との比較確認後、次に FinalStage の移行へ進む

### 完了条件

- 既存の Standard / FinalStage は壊れていない
- 新しいシミュレーション入口で固定対局の大会進行ができる
- 最小サンプル入力で CSV と表示を出力できる
- 旧 Standard 相当と新 Standard 相当を比較できる

## 大会最終状態の一般化出力の最小案

大会進行フレームワークでは、順位表とは別に、内部で使っている `TournamentMatchRecord` をそのまま外へ出せるようにする。

### 位置づけ

- `ResultCsv` / `ResultMarkdown` は順位表として残す
- それとは別に、一般化された大会最終状態テーブルを兄弟ファイルとして出力する
- 一般化テーブルのほうを、大会進行の基底表として扱う

### CSV の最小列

- `matchId`
- `stageId`
- `stageName`
- `firstPlayerId`
- `firstPlayerName`
- `secondPlayerId`
- `secondPlayerName`
- `startTime`
- `endTime`
- `status`
- `resultType`
- `roundNo`

ID 列だけだと人間には読みづらいので、初期段階では名前列も一緒に出す。

### Markdown の最小内容

- 概要
  - 総対局数
  - ステージ数
  - 完了対局数
  - 中止対局数
- 一覧表
  - CSV と同じ主要列を上位から順に表示する

Markdown は説明用の薄い出力に留め、正式な基底表は CSV とする。

### ファイル出力方針

- 既存の順位表 CSV / Markdown はそのまま出す
- 一般化結果テーブルは兄弟ファイルとして次のように出す
  - `tournament_match_records_*.csv`
  - `tournament_match_records_*.md`

## 旧 Standard と新基盤 Standard 相当の差分観点

新しい大会進行フレームワークを既存 Standard の代わりにしていくには、差分を次の観点で整理して埋めていく。

### 1. 入力

- 旧 Standard
  - 選手一覧 CSV
  - 対局 CSV または Round / First-Second / 対局記号表
  - 順位ルールセット
  - 同Elo対局時の先手勝率
- 新基盤 Standard 相当
  - Players CSV
  - Stages CSV
  - TournamentMatchRecords CSV
  - 任意 DSL
  - 同Elo対局時の先手勝率

差分としては、新基盤のほうが構造化されている代わりに、旧 Standard の入力をそのまま受けられない。

### 2. 結果決定

- 旧 Standard
  - 厳密計算または多数回シミュレーションで順位確率分布を出す
- 新基盤 Standard 相当
  - 現在は 1 回の大会実行で勝敗を確定して順位を 1 回だけ出している

ここが最大の差分で、旧 Standard の「確率分布」と、新基盤の「単発結果」は意味が違う。

### 3. 順位生成

- 旧 Standard
  - Neutral / Twill / TwillCommonOpp の順位ルールを適用する
- 新基盤 Standard 相当
  - 現在は `ByFinishedResultsRankingRule` で勝ち数ベースだけを見ている

つまり、新基盤はまだ Neutral 相当の簡略版に留まっていて、Twill 系が未移植。

### 4. 出力物

- 旧 Standard
  - 順位表 CSV / Markdown
- 新基盤 Standard 相当
  - 順位表 CSV / Markdown
  - 一般化結果テーブル CSV / Markdown

新基盤は出力物が広い一方で、順位表の中身はまだ旧 Standard と完全一致していない。

### 当面の埋め方

1. 旧 Standard と同じ入力ケースを新基盤でも表現できるようにする
2. 新基盤で複数試行の集計を回せるようにする
3. Neutral 相当の順位表を旧 Standard に近づける
4. その後で Twill 系順位ルールを移植する

### 最小4局ケースで見えた実差分

同じ 4 選手 4 対局のケースで確認したところ、差分は次のように現れた。

- 旧 Standard
  - 厳密計算で各選手の優勝確率と平均順位が 0〜100% の分布として出る
  - 例: Alice 26.41%、Bob 23.61%、Carol 29.57%、Dave 20.40%
- 新基盤 Standard 相当
  - 1 回の大会実行結果を順位表へ落としている
  - 優勝確率は 1 位になった選手だけ 100%、他は 0% になる
  - 平均順位も単発順位そのものになる

したがって、新基盤の次の不足点は明確で、**複数試行して順位分布を集計する層** がまだない。

## 新基盤の不足点（実装位置つき）

### 1. `TournamentEngine` は単発実行しか持っていない

- 現状: `TournamentEngine.Run()` は 1 回の大会を最後まで進めて `TournamentFrameworkExecutionResult` を 1 件返す
- 不足: 旧 Standard 相当へ近づけるには、これを複数回回して順位分布を合算する層が必要
- 候補位置:
  - `Application/TournamentFramework/TournamentEngine.cs` の外側に、複数試行用 runner / aggregator を追加する

### 2. `TournamentFrameworkExecutionResult` が単発結果しか持てない

- 現状: `FinalState`, `OverallRanking`, `TickCount`, `CompletedNaturally` のみ
- 不足: 複数試行の集計値を持つモデルがない
- 候補位置:
  - `Application/TournamentFramework` に `TournamentFrameworkAggregateResult` のような集計結果モデルを追加する

### 3. `BuildTournamentFrameworkCalculationResult()` が 1 回順位を 100% として落としている

- 現状: `SimulationTournamentFrameworkModeExecution.cs` で、得られた順位に対して `placeProbabilities[playerIndex, rank - 1] = 1.0` を入れている
- 不足: 旧 Standard に合わせるには、複数回の順位結果を集計して割合化する必要がある
- 候補位置:
  - `SimulationTournamentFrameworkModeExecution.cs` のこの補助関数を、単発結果ではなく aggregate result から組み立てる形に差し替える

### 当面の実装方針

1. `TournamentEngine` は単発 runner のまま維持する
2. その外側に `n` 回試行する aggregate 層を足す
3. aggregate 層から `CalculationResult` を構築する
4. その後に `ResultBuilders.cs` へ流す

## 実装後の確認メモ

- 実装: `SimulationTournamentFrameworkModeExecution.cs` に複数試行の aggregate 層を追加した
- 実装: `TournamentFrameworkModeContext` に `SimulationCount` を追加した
- 実装: STSAInput/2 の `Meta.SimulationCount` を大会進行フレームワークでも受け取れるようにした

入力:

- `Inputs/TournamentFramework/simulation_input_[大会進行フレームワーク_最小]_[STSAInput2].txt`
- 既定試行回数: `200,000`

### まだ残る差分

- 大会最終状態CSV/Markdownは代表実行 1 件の match records を出力しており、順位表だけが aggregate 結果になっている

### TournamentRuleSetMode の扱い

- 対話入力と STSAInput/2 の両方で `TournamentRuleSetMode` を指定できる
- aggregate の順位分布では、`Neutral / Twill / TwillCommonOpponentWeighted` を切り替えられる
- `TournamentFrameworkExecutionResult.OverallRanking` も `TournamentRuleSetMode` に応じた ranking rule で返すようにした
- 代表実行のコンソール順位表示も `OverallRanking` ベースで mode 対応にそろえた

### Twill 系の最小確認

- `Inputs/TournamentFramework/simulation_input_[大会進行フレームワーク_Twill_最小]_[STSAInput2].txt` を追加した
- `Inputs/TournamentFramework/simulation_input_[大会進行フレームワーク_Twill+CommonOpp_最小]_[STSAInput2].txt` を追加した
- 両方とも実行成功し、`計算方法` ラベルと `代表実行順位` 表示が mode 対応になることを確認した
- 最小 4 局ケースでは Twill と Twill+CommonOpp の分布値は同じになった
  - Alice `23.34%`
  - Bob `20.54%`
  - Carol `26.51%`
  - Dave `29.61%`

### Twill 系の差分確認用サンプル

- 既存の `先手8x後手8` 資産を TournamentFramework 向けに変換した
  - `Data/Players/players_[大会進行フレームワーク_先手8x後手8].csv`
  - `Data/Stages/stages_[大会進行フレームワーク_先手8x後手8].csv`
  - `Data/Matches/tournament_match_records_[大会進行フレームワーク_先手8x後手8].csv`
- 差分確認用の入力として次を追加した
  - `Inputs/TournamentFramework/simulation_input_[大会進行フレームワーク_先手8x後手8_Twill]_[STSAInput2].txt`
  - `Inputs/TournamentFramework/simulation_input_[大会進行フレームワーク_先手8x後手8_Twill+CommonOpp]_[STSAInput2].txt`
- `FirstPlayerWinRatePercent=70`、`SimulationCount=2,000` で両方を実行した
- aggregate の集計値はかなり近かった
  - 例: `飛` の優勝確率は Twill `19.07%`、Twill+CommonOpp `18.79%`
  - 例: `きりん` の平均順位は Twill `7.244`、Twill+CommonOpp `7.260`
- 一方で、代表実行順位では明確な差が出た
  - Twill の代表実行 1 位は `きりん`
  - Twill+CommonOpp の代表実行 1 位は `らいおん`
- つまり現状の大会進行フレームワークでも、最小 4 局ケースでは見えなかった `Twill` と `Twill+CommonOpp` の差を、より大きい固定対局サンプルでは確認できた
- ただし両実行とも 3 分の時間上限で打ち切られているため、分布差を厳密に見たいときは試行数や条件の追加調整が必要

### 同点処理について

- 単発の ranking 表示や代表実行は引き続き `ByFinishedResultsRankingRule` の順序確定を使う
- ただし aggregate の順位分布では、旧 Standard の Neutral と同様に、同点グループへ順位確率を分配するようにした
- これにより、固定対局の Standard 相当では、順位表の確率分布が旧 Standard により近い意味になる

これでどう（＾～＾）？　できそう（＾～＾）？


### きつい点

1.	現行計算エンジンが事前確定 Match 前提
新案は進行中にペアリングが増えるので、土台の前提が変わる。
2.	STSAInput/2 だけでは窮屈になる可能性が高い
一般化した大会ルールを key-value 羅列だけで書くのは厳しい。
3.	品質評価の再利用設計が必要
品質評価を含めないとしても、後で新基盤を大量試行する側になるので、
実行結果が毎回同じ形式で取れるようにしておく必要がある。


### 入力形式の見立て

ここは重要。
大会ルール本体は別ファイル化の方が安全。
おすすめは次。
* STSAInput/2
* 実行モード
* 選手CSV
* 対局記号表
* ルールファイルパス
* 乱数種
* 出力先
* 別ファイル
* 大会進行DSL

> 理由は、STSAInput/2 は「実行入力」に向いているが、
> 一般化された大会ルール記述は「構造定義」に近いから。

まったくその通りだぜ（＾～＾）


### 実現順のおすすめ

一気に全部は危ない。段階化が必要。
1.	新しい大会カーネルを作る
TournamentState, PairingRule, RankingRule, TerminationRule を定義
2.	現行 Standard を新基盤へ載せ替える
まず一番単純なラウンド制で検証
3.	現行 FinalStage を新基盤へ載せ替える
既存機能互換を取る
4.	その後にアリーナ制 / スイス式を追加する
この順なら、置き換え基盤として現実的。
