# 分析3ドメイン直列化構想

この文書は、現在の `Analysis` を将来的に次の3つの処理域へ分けるための設計メモです。

- シミュレーション域
- 最終順位付け域
- 大会品質評価域

現在の最上位フローは、`ApplicationWorkflow.Run` から見ると大きく次の2つです。

```text
開始
  -> 大会利用者域
  -> 分析
  -> 終了
```

この形は維持します。`Run` 直下へ細かい処理を戻さず、`分析` の内側を段階的に整理します。

## 目標形

最終的には、`分析` の中を次の直列処理として扱います。

```text
分析
  -> シミュレーション域
  -> 最終順位付け域
  -> 大会品質評価域
```

それぞれの処理域は、前段の結果を受け取り、次段へ渡せる結果オブジェクトを返します。

```text
AnalysisRequest
  -> SimulationDomain
      -> SimulationResult
  -> FinalRankingDomain
      -> FinalRankingResult
  -> TournamentQualityEvaluationDomain
      -> TournamentQualityReport
```

## 各処理域の責務

### シミュレーション域

大会ルール、選手、対局予定、先手勝率、試行回数などを受け取り、大会の結果候補や集計に使える中間結果を作ります。

現在の主な入口候補:

- `StandardSimulationMainline`
- `FinalStageSimulationMainline`
- `SimulationTournamentFrameworkMode`
- `SimulationEmptyMode`

将来は、これらを直接 `AnalysisRequestDispatcher` から呼び分けるだけでなく、`SimulationDomain` の入口へまとめます。

### 最終順位付け域

シミュレーション結果または入力済み対局結果から、最終順位を決めます。

現在は、シミュレーションと順位付けが密結合している箇所が残っています。将来は順位付けだけを独立させ、次のような中間結果を返せるようにします。

```text
FinalRankingResult
  - 順位
  - 選手別成績
  - タイブレーク情報
  - 品質評価へ渡す比較用データ
```

### 大会品質評価域

最終順位や入力済み対局結果をもとに、大会ルールの品質を評価します。

現在の主な入口候補:

- `TournamentQualityEvaluationMainline`
- `TournamentQualityEvaluationSingleRunExecutor`
- `TournamentQualityEvaluationSweepExecutor`

将来は、品質評価が必要な場合だけ `FinalRankingResult` と品質評価条件を受け取って実行します。

## 現在の実装との関係

現在の `AnalysisRequestDispatcher` は、`AnalysisStepRequest` の型を見て mainline を直接呼び分けています。

```text
AnalysisRequestDispatcher
  -> StandardSimulationRequest
  -> FinalStageSimulationRequest
  -> TournamentFrameworkSimulationRequest
  -> EmptySimulationRequest
  -> StandardQualityEvaluationRequest
  -> FinalStageQualityEvaluationRequest
```

これは移行途中の形として扱います。今後は、いきなり大きく作り替えるのではなく、次の順で分けます。

1. `AnalysisRequestDispatcher` からシミュレーション系の呼び分けを小さな dispatcher へ移す。
2. 品質評価系の呼び分けを小さな dispatcher へ移す。
3. シミュレーション mainline から順位付け結果を取り出せる形を探す。
4. `SimulationResult` / `FinalRankingResult` / `TournamentQualityReport` のような中間結果型を追加する。
5. `Analysis` 内部を3ドメイン直列の実行フローへ置き換える。

## 移行中に守ること

- `ApplicationWorkflow.Run` 直下は、`大会利用者域` と `分析` の2つを大きな入口として保つ。
- 要求ファイルの読み取り、手入力、要求ファイル書き出しは `大会利用者域` 側の責務に寄せる。
- 分析域では、入力を追加で尋ねる処理を増やさず、可能な限り `AnalysisRequest` から実行する。
- `InputFromSomewhere.UseText(...)` による対話入力偽装は、新ルートでは増やさない。
- 既存の legacy 入力ルートは、直通ルートへ移し終えるまで fallback として残す。
- 既存 smoke を壊さない。特に `STSAInput/4` の単一ステップと複数ステップは両方確認する。

## 次の小さな一手

大きな中間結果型を先に作るより、まず `AnalysisRequestDispatcher` の内部を分けます。

候補:

```text
AnalysisRequestDispatcher
  -> SimulationRequestDispatcher
  -> QualityEvaluationRequestDispatcher
```

この段階では、挙動は変えません。`ExecuteStandardSimulation` や `ExecuteFinalStageQualityEvaluation` を用途別のファイルまたは小クラスへ移し、あとで3ドメイン直列化しやすい形にします。
