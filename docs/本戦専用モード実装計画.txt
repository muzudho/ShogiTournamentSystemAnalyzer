【本戦専用モード】実装計画

目的
======
現行の ShogiTournamentPairingAnalyzer に、予選をシミュレーションせず、
本戦進出 16 ソフト（Apex 最大 8 / Innov 残り枠）だけを対象に順位分布を計算するモードを追加する。

このモードでは、次の特徴を扱う。
- 本戦進出した Apex は後手固定
- 本戦進出した Innov は先手固定
- 対局は Apex vs Innov のみ
- 総合順位は、まず本戦出場 Apex の順位、本戦不出場 Apex の順位、その後に本戦出場 Innov の順位が続く
- 本戦用の対局表に基づいて、順位分布や平均順位を計算する


このモードを先に作る理由
========================
1. 現行コードの資産を流用しやすい
   - Elo レーティング入力
   - 先手有利率入力
   - 対局表入力
   - モンテカルロシミュレーション
   - 結果CSV出力

2. 予選ルールの未確定部分を後回しにできる
   - 動的ペアリング
   - Apex 判定条件の細部
   - 二次予選の同点処理
   - 特別枠の扱い

3. まず面白い分析ができる
   - 上位 10 ソフトが団子状態
   - 上位 2 ソフトだけ突出
   - Apex 側の厚みがどれほど有利に働くか
   - Innov 側トップが全体 9 位に固定される制度の影響


想定する使い方
================
利用者は次の情報を入力する。

1. 先手有利率（%）
2. 本戦参加 16 ソフトの選手一覧CSV
3. 本戦の対局表
4. 各参加者が Apex か Innov かを示すグループ情報
5. 必要なら、本戦不出場 Apex の一覧または人数
5. 必要ならシミュレーション試行回数
6. 結果CSV出力先

本戦専用モードでは、通常の総合順位とは別に、
「本戦出場 Apex 内順位」「本戦出場 Innov 内順位」「制度上の総合順位」を計算する。


MVP（最小実装）
================
第1段階では、次だけ実装すればよい。

- モード選択を追加
  - 通常モード（現行）
  - 本戦専用モード（新規）

- 本戦参加者の入力
  - 既存の選手一覧CSVを流用
  - 追加で各参加者のグループを入力

- 本戦対局の入力
  - 既存の対局CSVまたは Round/Black-White/対局記号表を流用
  - ただし Apex vs Innov 以外の対局が含まれていないかを検証

- 結果計算
  - 各対局の勝敗シミュレーションは現行ロジックを流用
  - 集計時に Apex と Innov を別々に順位づけ
- 本戦出場 Apex の順位を先に並べる
- 本戦不出場 Apex がいれば、その順位を次に並べる
- その後ろへ本戦出場 Innov の順位を並べる

- 結果表示
  - コンソール表示
  - 結果CSV出力


設計方針
========
1. 既存モードを壊さない
   現行の総当たり戦モードはそのまま残す。
   新機能は分岐として追加し、既存利用者の操作を壊さない。

2. 対局シミュレーションと順位変換を分離する
   - 対局勝敗の計算
   - グループ別順位の計算
   - 制度上の総合順位への変換
   を別々の関数にする。

3. 入力形式はできるだけ既存資産を流用する
   学習コストを増やさない。

4. 予選込み大会シミュレーションへ拡張しやすくする
   将来、予選シミュレーションを足すときに再利用できる形にする。


追加が必要な概念
==================
現行コードに追加したい主要概念は次の通り。

1. グループ種別
   例:
   - Apex
   - Innov

2. 本戦参加者情報
   現在の Participant に加え、グループ情報を持たせるか、
   あるいは別の対応表を持つ。

   候補A: Participant に Group を直接持たせる
   候補B: Dictionary<string, FinalStageGroup> のような別管理にする

   MVP では、既存コードへの影響を減らすため、別管理の方が安全。

3. 制度上順位
   通常の勝数順位ではなく、
   - 本戦出場 Apex 内順位
   - 本戦出場 Innov 内順位
   - 本戦不出場 Apex の順位帯
   - 制度上総合順位
   の3種類を扱う。


入力仕様案
==========
案1: グループ対応CSVを別入力にする
--------------------------------
例:

group,name
Apex,"金"
Apex,"銀"
Innov,"桂"
Innov,"香"

利点:
- 既存の選手一覧CSVをそのまま流用できる
- 実装が分かりやすい
- 入力ミスの検出がしやすい

欠点:
- 入力が1つ増える


案2: 選手一覧CSVに group 列を追加する
--------------------------------------
例:

name,elo,group
"金",4990,Apex
"銀",4970,Apex
"桂",4950,Innov

利点:
- 入力ファイルが1つで済む

欠点:
- 現行の選手一覧CSV仕様を壊しやすい
- 通常モードとの兼ね合いが少し面倒


推奨
----
MVP では「案1: グループ対応CSVを別入力」を推奨。
理由は、既存の選手一覧CSV仕様に手を入れずに済むため。

補足:
現行ルールでは、本戦不出場 Apex も総合順位を Innov より先にもらう。
したがって、品質のよいシミュレーションを行うには、本戦出場 16 名だけでなく、
「本戦不出場 Apex が何名いて、どの Elo 帯にいるか」も将来は入力できるようにしたい。
ただし MVP では、まず本戦出場 16 名だけを対象にし、
本戦不出場 Apex は 0 名として扱う簡略版から始めるのが現実的である。


対局表の扱い
============
本戦専用モードでは、既存の対局入力形式を再利用する。

許可する入力形式:
- 対局CSV
- Round/Black-White/対局記号表

追加検証:
- Apex vs Apex の対局が含まれていないこと
- Innov vs Innov の対局が含まれていないこと
- Apex が後手側だけにいること（必要なら検証）
- Innov が先手側だけにいること（必要なら検証）

補足:
将来、定先戦専用の表形式を作ることもできるが、
MVP では既存入力形式の流用を優先する。


計算ロジック案
==============
全体の流れは次の通り。

1. 参加者一覧を読み込む
2. グループ対応表を読み込む
3. 対局表を読み込む
4. 入力検証を行う
   - 参加者数 16 名か
   - Apex が 8 名以下か
   - Innov が残り枠を埋めて 16 名になっているか
   - 名前対応が取れているか
   - 対局がすべてクロスグループか
5. 各対局の勝敗確率を Elo と先手有利率から計算
6. 厳密計算またはシミュレーションを実施
7. 各試行ごとに:
   - 各参加者の勝数を集計
   - Apex 内で順位づけ
   - Innov 内で順位づけ
   - 制度上総合順位へ変換
8. 全試行の分布を集計
9. 結果をコンソール表示・CSV出力


順位計算のポイント
==================
通常モードでは全参加者をひとまとめに順位づけしているが、
本戦専用モードでは次の2段階にする必要がある。

第1段階: グループ内順位
- 本戦出場 Apex を勝数で順位づけ
- 本戦出場 Innov を勝数で順位づけ
- 同点順位は現行と同様に等分配

第2段階: 制度上総合順位へ写像
- 本戦出場 Apex の順位 → 総合上位帯
- 本戦不出場 Apex がいれば、その直後へ固定的に挿入
- 本戦出場 Innov の順位 → その後ろの帯へ写像

このため、現行の AccumulatePlaceProbabilities 相当の関数を、
「グループ別順位対応版」として追加するのがよい。


必要になりそうな関数群
========================
例として、次のような関数分割が考えられる。

- ReadMode()
- RunStandardMode()
- RunFinalStageMode()
- ReadFinalStageGroupMap()
- TryParseFinalStageGroups()
- ValidateFinalStageParticipants()
- ValidateFinalStageMatches()
- CalculateFinalStageExactly()
- CalculateFinalStageBySimulation()
- AccumulateFinalStagePlaceProbabilities()
- BuildFinalStageResultRows()
- WriteFinalStageResultCsv()

全部を新設しなくてもよいが、
通常モードの関数を無理に流用しすぎると読みにくくなるため、
本戦専用の処理はある程度分けた方がよい。


結果表示案
==========
本戦専用モードでは、通常の列に加えて次の列が欲しい。

- group
- groupPlaceAverage
- overallPlaceAverage
- championshipProbabilityWithinGroup
- overallPlace1Probability（Apex のみ意味が強い）

ただし MVP では、列を増やしすぎない方がよい。

MVP 推奨列:
- participantName
- group
- originalElo
- effectiveElo
- eloDelta
- blackCount
- whiteCount
- blackWinRatePercent
- whiteWinRatePercent
- groupPlaceAverage
- overallPlaceAverage
- groupPlace1Probability
- overallPlace1Probability
- place1Percent ～ place16Percent

補足:
Apex 参加者は overallPlace1～8 にしか出現しないはずで、
Innov 参加者は overallPlace9～16 にしか出現しないはず。
この性質は検証にも使える。


コンソール操作案
================
起動直後にモードを選ばせる。

例:
1. 通常モード（現行の総当たり戦分析）
2. 本戦専用モード（Apex / Innov 定先戦分析）

本戦専用モードを選んだら、次の順で入力する。
- 先手有利率
- 選手一覧CSV
- グループ対応CSV
- 対局CSV または Round/Black-White/対局記号表
- 必要ならシミュレーション回数
- 出力先


検証ルール案
============
MVP では、次のエラー検出を入れたい。

- Apex と Innov 以外のグループ名を禁止
- 同じ選手が両グループに入っていないか
- 選手一覧にいない名前がグループ表に書かれていないか
- Apex が 8 名以下で、残りが Innov になっているか
- 対局に存在しない選手名がないか
- 同グループ同士の対局がないか
- Apex が先手側に現れていないか
- Innov が後手側に現れていないか

この検証が強いほど、シミュレーション結果の信頼性が増す。


段階的な実装順
==============
第1段階: モード分岐追加
- 起動時に通常モード / 本戦専用モードを選べるようにする
- 既存モードの挙動を維持する

第2段階: グループ入力追加
- グループ対応CSVの読み込み
- パース
- バリデーション

第3段階: 本戦入力検証追加
- Apex/Innov の人数確認
- クロスグループ対局のみ許可
- 定先条件の確認

第4段階: グループ別順位集計
- Apex と Innov を別々に順位づけ
- 制度上順位に変換
- 分布集計

第5段階: 表示・CSV出力
- コンソール列の整理
- 結果CSVヘッダー整備

第6段階: サンプルデータ整備
- Examples に本戦専用モード用の例を追加
- README に使用方法追記


テスト観点
==========
1. パース
- 正常入力
- グループ名の誤り
- 人数不足
- 重複名

2. バリデーション
- Apex vs Apex を弾く
- Innov vs Innov を弾く
- 定先条件違反を弾く

3. 計算
- 本戦出場 Apex 側の順位が overall 上位帯にしか出ない
- 本戦出場 Innov 側の順位が Apex 帯の後ろにしか出ない
- 確率総和が 100% になる

4. 退行確認
- 通常モードのビルドが通る
- 既存サンプル入力が壊れていない

おすすめの実装順（結論）
==========================
最初の1回は、次の範囲に絞るのがよい。

- 本戦専用モードを追加
- グループ対応CSVを別入力
- Apex 最大 8 / Innov 残り枠
- クロスグループ対局のみ対応
- グループ別順位から制度上総合順位へ変換
- コンソール表示とCSV出力まで対応

これなら、現行コードを大きく壊さず、
「このルールだとどんな順位分布になるか」を早く見られる。
