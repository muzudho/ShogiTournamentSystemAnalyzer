## 現在の判断

- 分析域の実行構造は、［シミュレーション域］、［最終順位付け域］、［大会品質評価域］の3つを直列に並べる方針。
- 内部の `AnalysisFlowSelection` は3大域プロパティで実行可否を表し、要求ファイル上の `AnalysisFlowSteps` は境界の互換表記として扱う。
- 将来的には、［標準ルール］/［本戦ルール］のような大分類をシステム内部の分岐軸から廃止する。ルールの違いは `RuleProfileMode.Standard` / `FinalStage` のラベルではなく、ルールプロファイルの細かな属性で表現する。
- 要求ファイルは `STSAInput/5` と `AnalysisFlowSteps` を現行形式として扱う。`STSAInput/4` は読み取り互換として継続し、`STSAInput/2` / `STSAInput/3` は `AnalysisFlowMode` 互換として読み取り継続する。

## 注意点

- PowerShell は最新版の `pwsh.exe` を使う。Windows PowerShell 5.1 (`powershell.exe`) は UTF-8 BOM 付きになりやすいため、テキストファイル作成・編集は PowerShell 7 系で行う。
- `apply_patch` が `writing outside of the project` で失敗するときは、[Codex apply_patch 回避メモ](./開発/Codex_apply_patch回避メモ_20260605.md) の PowerShell 手順を使う。
- Windows Defender が PowerShell reflection 検証を警戒したため、今後は [Windows Defender と Codex 実行メモ](./運用/WindowsDefenderとCodex実行メモ.md) に従う。DLL を PowerShell から `Assembly.LoadFrom` して private / internal メソッドを呼ぶ smoke は使わず、`dotnet run -- --smoke-request-writer-roundtrip ...` など通常の .NET 入口を使う。
- smoke 実行で `Output` 配下に CSV / Markdown / `.stsa.txt` が生成・更新される。実装差分から外すため、生成物は後始末する。
- staging は触らない。作業後に `git diff --cached --stat` が空であることを確認する。
- Codex チャットでは、巨大な tool output や Markdown のバッククォートを大量に含む出力で表示レイアウトが崩れ、その後応答が途切れることがある。調査時は `rg` の対象を絞り、Markdown/docs 全体や巨大ログをそのまま出さない。
