## 現在の判断

- 現在の `手入力保存` は `InputFromSomewhere.StartRecording()` でコンソール入力行を記録し、raw 入力ログとして書くだけで、`STSAInput/4` 仕様の要求ファイル生成にはなっていない。
- 手入力後に raw ログを STSAInput/4 へ変換するより、入力時点で `AnalysisRequest` / `AnalysisStepRequest` へ値を入れていき、実行も保存もそのオブジェクトを起点にする方針が自然。
- 保存時は `recordedLines` ではなく、`AnalysisRequest` を `STSAInput/4` テキストへシリアライズして書き出す Writer を本命にする。
- raw 入力ログが必要なら、STSAInput/4 要求ファイルとは別のデバッグログとして扱う。
- raw 手入力保存は要件外として、`InputFromSomewhere` の記録 API と未使用の raw 記録型を削除済み。
- `StsaInput4RequestWriter` を追加し、`AnalysisRequest` から `STSAInput/4` テキストを直接生成して保存する入口を追加済み。
- 手入力の `Simulation + Standard` / `FinalStage` / `TournamentFramework` / `Empty` は、入力時点で `AnalysisRequest` 化して `AnalysisRequestDispatcher` で実行する入口へ移行済み。
- 第73ステップの続きとして、手入力の `QualityEvaluation + Standard` / `FinalStage` も入力時点で `AnalysisRequest` 化して `AnalysisRequestDispatcher` で実行する入口へ移行済み。
- 手入力の品質評価では、実行前に既存の品質評価 reader で players / rule definition / input / execution options / output options を読み取り、`StandardQualityEvaluationRequest` または `FinalStageQualityEvaluationRequest` を作る。
- `AnalysisFlowSteps=Simulation,QualityEvaluation` 用のステップ別入力セクション仕様と実装を追加済み。パスカルケースのドットシンタックスで `Step.Simulation` / `Step.QualityEvaluation`、ステップ別入力は `Simulation.PlayersCsv` など、ステップ別出力は `Simulation.Output` / `QualityEvaluation.Output` を使う。共有入力セクションはステップ別入力がない場合の fallback として使う。
- 新規に要求ファイルや smoke 入力を作るときは、一番新しい仕様の表記に合わせる。例: Round 行列の記号対応見出しは `PlayerSymbols` を使い、旧見出し `Players` は既存ファイル互換の読み取り対象としてだけ扱う。

## 注意点

- `RequestFileCheckWorkflow` は raw `RequestText` を返す。`ApplicationWorkflow` 側で、まず `StsaInput4RequestParser.TryParse` を試し、未対応時だけ legacy converter を呼ぶ。
- `apply_patch` が `writing outside of the project` で失敗するときは、[Codex apply_patch 回避メモ](./開発/Codex_apply_patch回避メモ_20260605.md) の PowerShell 手順を使う。
- `STSAInput/4` の `AnalysisFlowSteps=Simulation,QualityEvaluation` は、`Step.Simulation` / `Step.QualityEvaluation` と共有またはステップ別入力セクションで要求ファイルから自動実行できる。
- 複数ステップ用の `Output` はステップ別セクション必須。共有 `Output` だけがある場合は parser がエラーにする。
- シミュレーションは `RuleProfileMode=Standard` / `FinalStage` ともに、20局以下なら厳密計算、20局超なら指定値または既定値 `200000` の近似計算として新ルートで動く。
- smoke 実行で `Output` 配下に CSV / Markdown / `.stsa.txt` が生成・更新される。
- smoke 実行で生成された `Output` 配下の CSV / Markdown は、実装差分から外すため削除済みにする。
- staging は触らない。作業後に `git diff --cached --stat` が空であることを確認する。
- Codex の1スレッドログ肥大が疑われる。最新の実測と新スレッド移行の暫定運用は [Codexログ出力量調査_20260606.md](./調査/Codexログ出力量調査_20260606.md) を参照する。
- Codex チャットでは、スレッド単位に限らず、巨大なログ出力や Markdown のバッククォートを大量に含む出力で表示レイアウトが崩れ、その後応答が途切れることがある。調査時は `rg` の対象を絞り、Markdown/docs 全体や巨大ログをそのまま出さない。
