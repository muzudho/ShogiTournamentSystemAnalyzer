## 注意点

- `RequestFileCheckWorkflow` は raw `RequestText` を返す。`ApplicationWorkflow` 側で、まず `StsaInput4RequestParser.TryParse` を試し、未対応時だけ legacy converter を呼ぶ。
- `apply_patch` が `writing outside of the project` で失敗するときは、[Codex apply_patch 回避メモ](./開発/Codex_apply_patch回避メモ_20260605.md) の PowerShell 手順を使う。
- Windows Defender が PowerShell reflection 検証を警戒したため、今後は [Windows Defender と Codex 実行メモ](./運用/WindowsDefenderとCodex実行メモ.md) に従う。DLL を PowerShell から `Assembly.LoadFrom` して private / internal メソッドを呼ぶ smoke は使わず、`dotnet run -- --smoke-request-writer-roundtrip ...` など通常の .NET 入口を使う。
- `STSAInput/5` の `AnalysisFlowSteps=Simulation,QualityEvaluation` は、`Step.Simulation` / `Step.QualityEvaluation` と共有またはステップ別入力セクションで要求ファイルから自動実行できる。`STSAInput/4` も互換入力として読み取れる。
- 複数ステップ用の `Output` はステップ別セクション必須。共有 `Output` だけがある場合は parser がエラーにする。
- シミュレーションは `RuleProfileMode=Standard` / `FinalStage` ともに、20局以下なら厳密計算、20局超なら指定値または既定値 `200000` の近似計算として新ルートで動く。
- smoke 実行で `Output` 配下に CSV / Markdown / `.stsa.txt` が生成・更新される。
- smoke 実行で生成された `Output` 配下の CSV / Markdown は、実装差分から外すため削除済みにする。2026-06-06 第114ステップの複数ステップ smoke 生成物は、`Remove-Item` は実行ポリシーでブロックされたが、明示パスの `.NET File.Delete` で後始末済み。
- 2026-06-06 第102ステップの TournamentFramework smoke 生成物 6 件は、削除コマンドが実行ポリシーでブロックされたため未追跡のまま残っている。実装差分ではない。
- staging は触らない。作業後に `git diff --cached --stat` が空であることを確認する。
- Codex の1スレッドログ肥大が疑われる。最新の実測と新スレッド移行の暫定運用は [Codexログ出力量調査_20260606.md](./調査/Codexログ出力量調査_20260606.md) を参照する。
- Codex チャットでは、スレッド単位に限らず、巨大なログ出力や Markdown のバッククォートを大量に含む出力で表示レイアウトが崩れ、その後応答が途切れることがある。調査時は `rg` の対象を絞り、Markdown/docs 全体や巨大ログをそのまま出さない。
- 2026-06-06 の強制終了後確認では、作業差分は破損していなかった。Codex 側の表示/ログ肥大による中断が本命で、アプリ側なら `Program.cs` が `OperationCanceledException` 以外を捕捉しない点に注意する。
- 2026-06-06 16:00頃の追加確認では、13:10 以降の大きめ `.jsonl` は最大約 656.4 KiB まで JSONL として正常完結していた。ログファイル破損ではなく、チャット表示または拡張機能側が大量 tool output で途切れる線が濃い。
