# プログラム4パート構成 第135ステップ実行計画

## 目的

手入力時の「シミュレーションをするか」「品質評価をするか」の質問位置を、実際に各処理域へ入る直前へ移す。

現状は［大会利用者域］の中で分析フローを先に確定してから、［要求ファイル］保存や後続の［シミュレーション域］・［大会品質評価域］へ進んでいる。そのため、入力フローが実行フローより先走って見える。

手入力フローを次の順序へ寄せる。

```text
開始
↓
大会利用者域
↓
要求ファイル準備
↓
シミュレーション域の直前で実行可否を確認
↓
最終順位付け域
↓
大会品質評価域の直前で実行可否を確認
↓
実行した手入力ステップを要求ファイルとして保存
↓
終了
```

## 実施内容

1. `ApplicationTournamentUser` の手入力責務を縮める。
   - コマンドライン引数の要求ファイル指定確認は維持する。
   - 要求ファイルチェックと parser 直通要求の生成は維持する。
   - 手入力時は、分析フロー・ルールプロファイル・各処理域入力をここで確定しない。
   - 手入力の要求ファイル保存先だけを先に確認する。
2. `TournamentUserDomainResult` を見直す。
   - 要求ファイル指定ありの場合は、既存通り `AnalysisRequest` を持つ。
   - 手入力の場合は、手入力モードであることと要求ファイル保存先を持つ。
   - 手入力時点では `AnalysisFlowSelection` と `RuleProfileAttributes` を既定値で確定した扱いにしない。
3. 手入力用の実行状態を追加する。
   - 実際に実行した `AnalysisStepRequest` を順に蓄積できるようにする。
   - 後で `AnalysisRequest` と `AnalysisFlowSelection` を組み立てられる形にする。
4. ［シミュレーション域］の入口で、手入力時だけ実行可否を尋ねる。
   - `シミュレーションをしますか？` をここで聞く。
   - `はい` の場合だけ、シミュレーション用の `RuleProfileAttributes` と入力を読む。
   - 読み取った request を実行し、手入力用の実行状態へ記録する。
5. ［大会品質評価域］の入口で、手入力時だけ実行可否を尋ねる。
   - `品質評価をしますか？` をここで聞く。
   - `はい` の場合だけ、品質評価用の `RuleProfileAttributes` と入力を読む。
   - 品質評価では `PairingSource=ScheduledMatches` 制約をこの段階で説明する。
   - 読み取った request を実行し、手入力用の実行状態へ記録する。
6. `ManualAnalysisRequestReader` の入口を分割する。
   - シミュレーション request 読み取りを、シミュレーション域から呼べる入口にする。
   - 品質評価 request 読み取りを、大会品質評価域から呼べる入口にする。
   - 既存の単一ステップ前提 `TryRead(...)` は削除または互換 wrapper に縮小する。
7. 手入力の要求ファイル書出位置を実行後へ移す。
   - 実際に実行した step だけを `AnalysisRequest` に入れる。
   - 保存先が指定されている場合だけ `StsaInputRequestWriter.BuildAttributeLines(...)` で `STSAInput/5` を書く。
   - 何も実行しなかった場合は、要求ファイルを書かずに終了理由を表示する。
8. コンソールガイド表示を調整する。
   - `ProgramConsoleGuide.PrintSelectedMainline(...)` は要求ファイル指定ありの直通実行で使う。
   - 手入力では主線が確定したタイミング、または各域の直前に必要な案内を表示する。
   - `■［分析］` の単独表示は、各処理域の表示へ寄せる。

## 変えないこと

- 要求ファイル指定ありの `AnalysisRequestDispatcher.Execute(...)` 直通実行。
- `STSAInput/5` の parser / writer 形式。
- `STSAInput/4` の読み取り互換。
- シミュレーション、最終順位付け、品質評価の計算ロジック。
- CSV / Markdown の出力内容。

## 実装上の注意

- 要求ファイル指定ありでは、要求ファイル内の `Steps` を信頼し、追加質問しない。
- 手入力では、実行しない処理域の入力を読まない。
- シミュレーションと品質評価で別々の `RuleProfileAttributes` を選べる構造にする。将来、複数 step の要求ファイル保存にそのままつながるため。
- 最終順位付け域は現状シミュレーション域の中から呼ばれているため、今回の変更では入口位置だけ維持し、処理分離は別ステップに回す。
- 要求ファイル保存は「入力ログ」ではなく「実行した要求の再現ファイル」として扱う。

## 検証

実装後に次を確認する。

- `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj`
- 要求ファイル指定ありの既存 smoke 実行。
- 手入力でシミュレーションだけ実行し、必要な質問がシミュレーション域直前に出ること。
- 手入力で品質評価だけ実行し、必要な質問が大会品質評価域直前に出ること。
- 手入力でシミュレーションと品質評価を両方実行し、実行順と質問順が一致すること。
- 手入力でどちらも実行しない場合、不要な入力を読まずに終了すること。
- 手入力から保存した `STSAInput/5` 要求ファイルを `--request-file` で再実行できること。
- `git diff --check`

## 次ステップ候補

- 実装後、手入力の対象ルール選択を属性直接入力・ルールプロファイルファイル選択へ拡張する。
- 最終順位付け域をシミュレーション域から分離し、アプリケーション直下の［最終順位付け域］へ実処理を移す。
- 手入力で複数 step を作る場合の確認表示を、`ProgramConsoleGuide` に集約する。
