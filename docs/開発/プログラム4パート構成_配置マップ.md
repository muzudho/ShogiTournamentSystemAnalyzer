# プログラム4パート構成 配置マップ

## 作業方針

この作業では、先に `Application/Execution` 配下へライフサイクル上のフォルダーを切り、コードをそのフォルダーへ配置しながら進める。

メソッド単位の切り出しだけを続けると全体の進捗が見えにくくなるため、以後は「どのパートへ置いたか」を優先して見える化する。

## 移動ルール

1. まず既存の責務が明確なクラスから移動する。
2. namespace は最初から大きく変えず、ビルドが通る単位で進める。
3. 移動したら、この表の配置候補と実コードの差を小さくする。
4. 1 ステップごとにビルドとスモーク確認を行う。
5. `#メモ：` や `#ここは：` で始まるファイルはユーザー用の目印として扱い、移動や編集対象から外す。
6. `Shared` は多用しない。複数パートにまたがり、単一パートへ置くと誤解が増えるものだけに使う。

## 直下に残すもの

`Application/Execution` 直下には、ライフサイクルの各パートを束ねるアプリケーション実行ルートだけを残す。

- `ApplicationRun.cs`
- `ApplicationRequestRun.cs`
- `ApplicationRequestSessionRun.cs`

これらはパートそのものではなく、パート間を接続する実行順序を表す。

## Analysis 配下の 4大域 / 4大境界

`Application/Execution/Analysis/` 配下では、4大域と4大境界をフォルダーで分ける。

- `Domains/TournamentUser/`
- `Domains/Simulation/`
- `Domains/Ranking/`
- `Domains/TournamentQualityEvaluator/`
- `Boundaries/Request/`
- `Boundaries/TournamentFinalState/`
- `Boundaries/FinalRanking/`
- `Boundaries/TournamentQualityReport/`

第40ステップでは、分析以降の既存実行ファイルをまず 4大域側へ配置する。境界側は、データ受け渡しの責務が明確になった段階で移す。
