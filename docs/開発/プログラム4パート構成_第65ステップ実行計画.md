# プログラム4パート構成 第65ステップ実行計画

## 目的

［分析］フェーズの入り口を、対話入力列ではなく［要求オブジェクト］に寄せる。

現在は、要求ファイルを読んだあとも `InputFromSomewhere.UseText(requestText)` で対話入力を偽装し、分析フェーズ内の `ConsolePromptReaders` / `ConsoleInputReaders` がそれを読む構造になっている。

第65ステップでは、この構造を最終形へ移すための最初の実装スライスを作る。

最終形:

```plaintext
要求ファイル or 手入力
  ↓
要求テキスト読込
  ↓
要求テキスト
  ↓
要求テキストParser
  ↓
要求オブジェクト
  ↓
分析フェーズ
```

分析フェーズでは、原則として［要求テキスト］も `InputFromSomewhere` も読まない。

## 現状の問題

現在の要求ファイル入力は、次の流れになっている。

```plaintext
STSAInput/4
  ↓
StsaInputLegacyConverter
  ↓
対話入力行の文字列
  ↓
InputFromSomewhere.UseText(...)
  ↓
分析フェーズ内の入力 reader
```

このため、要求ファイル読込フェーズと分析フェーズは見た目上は分かれていても、実体はまだ強く結合している。

特に問題になる点:

- 分析フェーズの中で `InputFromSomewhere.ReadLine()` が呼ばれる。
- 要求ファイル converter が、分析フェーズ内のプロンプト順を正確に知っている必要がある。
- 対局数によって `SimulationCount` の消費有無が変わるなど、入力行ずれを起こしやすい。
- 複数ステップ `AnalysisFlowSteps=Simulation,QualityEvaluation` を実装しようとすると、ステップ別の入力列連結が複雑になる。

## 第65ステップのゴール

第65ステップでは、いきなり全分析モードを置き換えない。

まず次の骨組みを追加する。

- `RequestText` 型
- `AnalysisRequest` 型
- `AnalysisStepRequest` 系の型
- `StsaInput4RequestParser`
- `AnalysisRequestDispatcher` または `AnalysisFlowDispatcher` の `AnalysisRequest` 受け取り口

そして、最初の対応モードを 1 つに絞る。

推奨する最初の対象:

- `AnalysisFlowSteps=Simulation`
- `RuleProfileMode=Standard`
- 20 局以下の厳密計算ケース

理由:

- 既に `StandardSimulationMainline.Run(StandardModeSimulationContext)` は context を受け取れる。
- `StandardModeSimulationContext` は要求オブジェクトから構築しやすい。
- 20 局以下なら `SimulationCount` の追加入力を考えなくてよい。
- 第64で追加した `simulation_input_[先手1x後手1]_[Neutral_Exact]_[STSAInput4_smoke].txt` をそのまま検証に使える。

## 用語

### 要求テキスト

要求ファイルまたは手入力から作られる、STSAInput 形式のテキスト。

候補型:

```csharp
internal sealed record RequestText(
    string FormatName,
    IReadOnlyList<string> Lines,
    string? SourcePath);
```

### 要求オブジェクト

分析フェーズに渡す、解析済みの要求。

候補型:

```csharp
internal sealed record AnalysisRequest(
    AnalysisFlowSelection FlowSelection,
    RuleProfileMode RuleProfileMode,
    IReadOnlyList<AnalysisStepRequest> Steps);
```

### 分析ステップ要求

分析ステップごとの具体要求。

候補型:

```csharp
internal abstract record AnalysisStepRequest;

internal sealed record StandardSimulationRequest(
    TournamentRuleSetMode TournamentRuleSetMode,
    double FirstPlayerWinRatePercent,
    IReadOnlyList<Player> Players,
    IReadOnlyList<Match> Matches,
    int? SimulationCount,
    string? OutputPath) : AnalysisStepRequest;
```

第65では `StandardSimulationRequest` だけでもよい。

## 実装方針

### 1. 既存 legacy converter はすぐ消さない

`StsaInputLegacyConverter` は第65では残す。

新しい parser と dispatcher を横に追加し、最初の smoke が通ったあと、対応済みモードだけ新ルートへ切り替える。

### 2. 分析フェーズの新入口を追加する

現在:

```csharp
AnalysisFlowDispatcher.Execute(requestBoundary.AnalysisFlowSelection, requestBoundary.RuleProfileMode);
```

新入口候補:

```csharp
AnalysisRequestDispatcher.Execute(analysisRequest);
```

または既存 dispatcher に overload を追加する。

```csharp
AnalysisFlowDispatcher.Execute(AnalysisRequest request);
```

第65では、新クラス `AnalysisRequestDispatcher` を推奨する。既存の `AnalysisFlowDispatcher` は対話入力ルートとして残し、差分を読みやすくする。

### 3. 出力先プロンプトを分離する

`StandardSimulationMainline.Run(context)` は context を受け取れるが、出力先はまだ `ResolveFinalRankingOutputPaths(...)` 内で `ConsolePromptReaders.ReadTextWithDefault(...)` を読む。

第65の最小対応では、次のどちらかを選ぶ。

推奨:

- `StandardSimulationMainline` に `Run(context, outputPathOverride)` のような overload を追加する。
- 既存 `Run(context)` は手入力互換として残す。
- 要求オブジェクトルートだけ `outputPathOverride` を渡す。

これで、少なくとも新ルートの標準シミュレーションでは分析フェーズ内の出力先入力を読まなくなる。

### 4. RequestBoundary は当面薄いままにする

`RequestBoundary` をいきなり全データ保持クラスへ太らせない。

第65では、`RequestBoundary` に `AnalysisRequest? AnalysisRequest` を追加する程度に留める。

候補:

```csharp
public AnalysisRequest? AnalysisRequest { get; set; }
```

ただし `Domain.Request` から `Application` の request 型へ依存させたくない場合は、`ApplicationWorkflow` 内のローカル変数として持つ。

推奨は、まず `ApplicationWorkflow` のローカル変数で持つこと。

## 作業順

1. `Application/RequestParsing` などの名前空間を作る。
2. `RequestText` を追加する。
3. `AnalysisRequest` / `AnalysisStepRequest` / `StandardSimulationRequest` を追加する。
4. `StsaInput4RequestParser` を追加し、`STSAInput/4` の `Simulation + Standard` だけ解析する。
5. parser は既存の `StsaInputSectionParser` と `InputParsers` を再利用する。
6. `AnalysisRequestDispatcher` を追加し、`StandardSimulationRequest` だけ実行する。
7. `StandardSimulationMainline` に、要求オブジェクトルート用の出力先指定 overload を追加する。
8. `ApplicationWorkflow` の要求ファイル入力側で、新 parser が対応できる場合だけ新ルートを使う。
9. 未対応モードは既存 legacy converter ルートへフォールバックする。
10. 第64で追加した標準シミュレーション smoke を、新ルートで通す。

## ApplicationWorkflow の暫定形

第65完了時点の理想は、要求ファイル入力だけ次のようにすること。

```plaintext
RequestFileCheckWorkflow
  ↓
RequestText
  ↓
StsaInput4RequestParser.TryParse
  ├─ 成功: AnalysisRequest を RunAnalysisDomain へ渡す
  └─ 未対応: 既存 legacy input text ルートへフォールバック
```

手入力は第65では既存のままでもよい。

理由:

- 手入力を先に STSAInput/4 テキスト化するには、入力保存仕様の整理が必要。
- 第65の焦点は、分析フェーズ内で要求テキストを読まない新ルートを作ること。

第65では、全 smoke を新ルートへ乗せる必要はない。

- 手入力フローの全面改修

- 複数ステップ `Simulation,QualityEvaluation` の完全対応
- `TournamentFramework` / `Empty` の新ルート対応
- 品質評価モードの新ルート対応

## 完了条件

第65ステップ完了と言える条件:

- `STSAInput/4` の標準シミュレーション smoke が、対話入力偽装ではなく `AnalysisRequest` ルートで実行される。
- その新ルートでは、分析フェーズ内で `InputFromSomewhere.ReadLine()` を読まない。
- 未対応モードは既存ルートで動き続ける。
- `dotnet build` が成功する。
