# プログラム4パート構成 第64ステップ実装計画

## 目的

現在の `ApplicationWorkflow.cs` に合わせて、要求ファイル入力から分析フローを安定して実行できるようにする。

古い第64ステップ計画は、`ApplicationWorkflow` 内に旧実行ルートと新4パート構成スケルトンが同居している前提だった。しかし現在は、`Program.Main` が `ApplicationWorkflow.Run(args)` を呼び、`ApplicationWorkflow` が次の流れを直接担当している。

1. `Opening`
2. `RunTournamentUserDomain`
3. `RunAnalysisDomain`
4. `WriteRequestFile`

したがって、第64ステップでは `LegacyApplicationWorkflow` や `FourPartApplicationWorkflow` を新設しない。現状の本当の未解決点である、`STSAInput/4` の要求ファイル変換と分析ステップ実行の接続を整理する。

## 現状コードの要点

- `ApplicationWorkflow.Run` は `RequestBoundary` を作り、要求ファイルまたは手動入力を通して `AnalysisFlowSelection` と `RuleProfileMode` を決める。
- 手動入力では `ConsolePromptReaders.ReadAnalysisFlowSelection()` が「シミュレーションをしますか？」「品質評価をしますか？」を順に聞く。
- `AnalysisFlowDispatcher.Execute` は `flowSelection.Steps` を順番に実行できる。
- 要求ファイル入力では `RequestFileCheckWorkflow` が `STSAInput/4` を対話入力行へ変換し、`ConsoleInput.UseText(requestText)` で後続処理へ流し込む。
- `StsaInputLegacyConverter` は `flowSelection.Steps.Count != 1` のとき中断する。
- `StsaSimulationLegacyConverter` は現在、`TournamentFramework` と `Empty` だけを変換できる。
- `Standard` / `FinalStage` のシミュレーションは対話入力 reader で動くが、要求ファイル converter は未実装。
- シミュレーションは対局数が 20 局以下なら厳密計算、21 局以上なら試行回数を追加で聞く。ここを考慮しないと要求ファイルの行ずれが起きる。

## 第64ステップの実装ゴール

第64ステップでは、まず「要求ファイルを対話入力行へ変換する現行方式」を維持したまま、標準・本戦シミュレーションの `STSAInput/4` 変換を追加する。

複数ステップ `AnalysisFlowSteps=Simulation,QualityEvaluation` は、第64ステップでは仕様と変換構造を準備する。ただし、実行確認の主対象は単一ステップの `Simulation` とし、複数ステップの完全対応は第65ステップ候補に回す。

理由は、複数ステップ対応には、`Simulation` 用入力列と `QualityEvaluation` 用入力列を 1 つの要求ファイル内で衝突なく持つためのセクション設計が必要だから。

## 実装方針

### 1. ApplicationWorkflow は最小限だけ触る

`ApplicationWorkflow.cs` は実行ルート分離しない。ただし、要求ファイル入力では `InputFromSomewhere.UseText(requestText)` 後に、変換済み入力列の先頭から `AnalysisFlowSelection` と `RuleProfileMode` を `RequestBoundary` へ読み込む必要がある。

これをしないと、要求ファイルの `AnalysisFlowSteps=QualityEvaluation` でも `RequestBoundary` の既定値である `Simulation / Standard` が実行される。

### 2. STSAInput/4 の単一 Simulation 対応を追加する

`StsaInputLegacyConverter` のこの未対応分岐を解消する。

```csharp
if (analysisFlowMode != AnalysisFlowMode.QualityEvaluation) throw new OperationCanceledException(...);
```

追加対象:

- `AnalysisFlowSteps=Simulation`
- `RuleProfileMode=Standard`
- `RuleProfileMode=FinalStage`

既存の `TournamentFramework` と `Empty` 変換は維持する。

### 3. シミュレーション用 legacy 入力行を正確に作る

`Standard` の対話入力順:

1. 分析フロー選択 prefix
2. ルール選択 prefix
3. `TournamentRuleSetMode`
4. `FirstPlayerWinRatePercent`
5. `PlayersCsv` セクション、空行終端
6. `MatchesInput` セクション、`END` 終端
7. 対局数が 21 局以上の場合だけ `SimulationCount`
8. `OutputPath` または `SummaryOutputPath`

`FinalStage` の対話入力順:

1. 分析フロー選択 prefix
2. ルール選択 prefix
3. `FirstPlayerWinRatePercent`
4. `PlayersCsv` セクション、空行終端
5. `GroupMapCsv` セクション、空行終端
6. `AdditionalApexPlayersCsv` セクション、空行終端
7. `AdditionalApexPlacementMode`
8. `BoundaryRescueMode`
9. `MatchesInput` セクション、`END` 終端
10. `ReferenceMatchesInput` セクション、`END` 終端
11. 対局数が 21 局以上の場合だけ `SimulationCount`
12. `OutputPath` または `SummaryOutputPath`

`VariableTop8Mode` は品質評価側にはあるが、現在の本戦シミュレーション reader には無い。シミュレーション変換へ混ぜない。

### 4. 対局数判定は converter 側で既存 parser を使う

試行回数行を出すかどうかは、実行時と同じ 20 局閾値に合わせる。

実装案:

- `PlayersCsv` を `InputParsers.TryParsePlayers` で読む。
- `MatchesInput` を `InputParsers.TryParseMatches` で読む。
- パースできない場合は `OperationCanceledException` で要求ファイルチェック時に止める。
- `matches.Count > 20` の場合だけ `SimulationCount` を legacy 入力行へ追加する。
- 20 局以下なのに `SimulationCount` が指定されていても、行ずれ防止のため入力行へは出さない。警告表示は第64では追加しない。

### 5. converter の構造を段階的に整理する

いきなり直接モデル化へ移行しない。

第64で追加する候補:

- `StsaSimulationLegacyConverter.ConvertStandard`
- `StsaSimulationLegacyConverter.ConvertFinalStage`
- `StsaSimulationLegacyConverter.AppendSimulationCountIfApproximationNeeded`
- `StsaSimulationLegacyConverter.ParsePlayersForPromptShape`
- `StsaSimulationLegacyConverter.ParseMatchesForPromptShape`

`StsaQualityEvaluationLegacyConverter` は既存 smoke が通っているため、第64では大改造しない。

## STSAInput/4 仕様更新

`docs/設計/STSAInput4仕様.md` に、単一 `Simulation` の対応範囲を追記する。

最小例を追加する。

- `AnalysisFlowSteps=Simulation`
- `RuleProfileMode=Standard`
- `TournamentRuleSetMode=Neutral`
- `FirstPlayerWinRatePercent=51`
- `SimulationCount=10`
- `PlayersCsv`
- `MatchesInput`
- `Output`

注意書きも追加する。

- `SimulationCount` は対局数 21 局以上の近似計算時だけ使われる。
- 対局数 20 局以下では厳密計算になり、`SimulationCount` は入力行へ流さない。
- `AnalysisFlowSteps=Simulation,QualityEvaluation` は引き続き予約仕様で、第64では完全自動実行の対象外。

## 作業順

1. `StsaSimulationLegacyConverter` に `Standard` / `FinalStage` 変換メソッドを追加する。
2. `InputParsers` を使って、converter 側で players / matches を検証し、対局数を得る補助メソッドを作る。
3. `StsaInputLegacyConverter` の分岐を更新し、`Simulation + Standard` と `Simulation + FinalStage` を新メソッドへ流す。
4. `docs/設計/STSAInput4仕様.md` に単一シミュレーション仕様を追記する。
5. `Inputs/Smoke` に `STSAInput/4` の単一シミュレーション smoke を追加する。
6. `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj` を実行する。
7. 既存の品質評価 smoke を実行して、既存変換が壊れていないことを確認する。
8. 新規シミュレーション smoke を実行する。

## 第64ステップでやらないこと

- `ApplicationWorkflow` の実行ルート分離。要求ファイル入力の `RequestBoundary` 反映だけは第64で行う
- `LegacyApplicationWorkflow` / `FourPartApplicationWorkflow` の新設
- 要求ファイルを内部モデルへ直接変換する全面改修
- `AnalysisFlowSteps=Simulation,QualityEvaluation` の完全自動実行
- `TournamentFramework` / `Empty` と `QualityEvaluation` の組み合わせ対応
- `VariableTop8Mode` を本戦シミュレーションへ接続すること

## 第65ステップ候補

第65ステップでは、複数ステップ要求ファイルの仕様を実装する。

候補仕様:

```plaintext
#[Section] Simulation.Meta
...
#[EndSection]

#[Section] QualityEvaluation.Meta
...
#[EndSection]
```

または、共通データを上位セクションに置き、ステップ別に差分だけを持つ。

第65で必要になる作業:

- `Simulation` 用 body と `QualityEvaluation` 用 body を prefix なしで生成できるようにする。
- `StsaInputLegacyConverter` が prefix を 1 回だけ生成し、ステップ順に body を連結する。
- `Simulation` と `QualityEvaluation` が同じ `PlayersCsv` / `MatchesInput` を使う場合と、別セクションを使う場合のルールを決める。
- 複数ステップ smoke を追加する。

## フリーズ対策

フリーズ原因はコードから断定しない。ただし、今回の計画では作業範囲を `StsaInputLegacyConverter` と `StsaSimulationLegacyConverter` に絞る。

`ApplicationWorkflow` の大規模分離や、直接モデル化への全面移行は行わない。これにより、実装中に入口構造と要求ファイル変換の両方を同時に変えるリスクを避ける。
