# プログラム4パート構成 第64ステップ実行計画

## 目的

`ApplicationWorkflow` に同居している旧フレームワークと新4パート構成を、名前付きの実行ルートへ分離する。

第63ステップでは、同居状態を解消し、最上位入口を薄くする方針を立てた。第64ステップでは、その方針を実装し、旧フレームワークから新フレームワークへ移行するための入口を明確にする。

## 対象

- `ShogiTournamentSystemAnalyzer/Application/ApplicationWorkflow.cs`
- `ShogiTournamentSystemAnalyzer/Application/LegacyApplicationWorkflow.cs`
- `ShogiTournamentSystemAnalyzer/Application/FourPartApplicationWorkflow.cs`

## 方針

### 1. 旧フレームワークを退避する

旧フレームワークは `LegacyApplicationWorkflow` として残す。

```csharp
ApplicationStartup.Start();
RequestWorkflow.Run(args);
```

これは旧導線の比較用・退避用ルートとして扱う。

### 2. 新4パート構成を名前付きルートへ移す

`RequestBoundary` を作り、次の順に接続する導線を `FourPartApplicationWorkflow` へ移す。

1. `RequestFileCheck`
2. `ManualInput`
3. `RequestFileCreate`
4. `Analysis`

### 3. `ApplicationWorkflow` を薄い入口に戻す

`ApplicationWorkflow.Run(args)` は、本番ルートを選んで呼ぶだけにする。

現時点のコードでは4パート構成スケルトンが実際に実行されていたため、第64ステップではその状態を明示化し、`FourPartApplicationWorkflow.Run(args)` を呼ぶ。

## 確認

- `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj`

## 第64ステップで実装しないこと

- 旧 `RequestWorkflow` の内部構造の大幅変更
- 新4パート構成の各節点の本格実装
- 入力ファイル仕様や出力仕様の変更
