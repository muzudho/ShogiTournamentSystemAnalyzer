# プログラム4パート構成 第62ステップ実行計画

## 目的

シミュレーションの最終順位表を、`Standard` と `FinalStage` の2系統ストラテジーとして増やすのではなく、`共通部分` と `自由形式部分` に分けて一般化できるか調査し、次の実装方針を決める。

## 結論

一般化は可能。

現状コードにもすでにその形の前兆がある。`SimulationResultRowCommonData` が共通部分を持ち、`StandardResultRow` と `FinalStageResultRow` がそれぞれ固有列を持っている。ただし、現在は固有列が型そのものに固定されているため、利用側では結局 `StandardResultRow` / `FinalStageResultRow` の具象型分岐が残っている。

第62ステップ以降では、型をルール名で分ける方向ではなく、次の構造へ寄せる。

- 共通部分: プレイヤー名、レーティング、対局数、先後勝率、順位確率など、どのルールでも使う基礎データ
- 自由形式部分: 優勝確率、平均順位、グループ、グループ内1位確率、本戦総合順位など、ルールやレポートに応じて増減する列と表示部品

## 一般化の基本方針

むずでょ原文の一般化案に合わせて、以後は次の3レイヤーで考える。

- 入力側: 入力ファイルや大会ルール定義の書式を知っている。外部データを内部用の共通モデルへ変換する。
- 内部側: 入力書式や出力書式を知らない。プレイヤー、対局、タイム、状態、対局記録などの最小限の共通概念だけで計算する。
- 出力側: 出力する表、列、Markdown セクション、CSV スキーマを知っている。内部結果を表示用の共通部分と自由形式部分へ変換する。

この方針では、`Standard` / `FinalStage` の違いを内部モデルの型分岐として増やさない。違いは入力側の変換、将来の DSL 解釈、出力側の report profile に寄せる。

## 将来の DSL 化方針

将来的には、プログラム本体が［標準ルール］か［本戦ルール］かを知っている状態を避ける。

［依頼という境界］の入力ファイルで、大会ルール、プレイヤー一覧、順位付け設定などの条件を組み合わせる。さらに、大会内の振る舞いは外部ファイルの DSL に書き、入力側がそれを読み取って内部側が扱える共通モデルや実行計画へ変換する。

この方針では、内部側の `if Standard` / `if FinalStage` を増やさない。内部側は、DSL や入力ファイルの名前を知らず、解釈済みのルール、ペアリング条件、終了条件、順位付け条件だけを受け取って振る舞う。

短期的には `RuleProfileMode` は既存入力互換とフロー選択のために残す。ただし、設計上の最終目標は、`RuleProfileMode` を内部ロジックの種類判定ではなく、入力側が DSL や設定セットを選ぶための互換ラベルへ縮退させること。
## 調査結果

### Domain

`ShogiTournamentSystemAnalyzer/Domain/Simulation/SimulationModels.cs` では、すでに次の分離がある。

- `SimulationResultRowCommonData`
- `StandardResultRow`
- `FinalStageResultRow`
- `ISimulationResultRow`

これは「共通部分 + 固有部分」に近いが、固有部分が `StandardResultRow` / `FinalStageResultRow` という型に焼き付いている。今後ルールが増えると、行型と分岐が増えやすい。

### Application

`SimulationMainlineExecutionResult<TRow>` は行型を generic にしているが、`StandardSimulationMainline` と `FinalStageSimulationMainline` では `StandardResultRow` / `FinalStageResultRow` を直接扱っている。

特に `FinalStageSimulationMainline` は、グループ分け Off の場合に `StandardResultRow`、On の場合に `FinalStageResultRow` を返すため、同じルールプロファイル内でも出力行型が分岐している。

ここは「本戦ルールだから FinalStageResultRow」ではなく、「この実行結果にはどの自由形式列が付くか」という形に変える候補になる。

### CSV 出力

`FinalRankingCsvFileWriter` は `ISimulationResultRow.GetFinalRankingCsvSpecificColumns()` を呼び出しており、外側の処理はかなり共通化されている。

一方、列定義ファイルは次の2つに分かれている。

- `Data/Definitions/FinalRankingStandardTableType.json`
- `Data/Definitions/FinalRankingFinalStageTableType.json`

この2ファイルも、内容としては「共通列 + 固有列」の組み合わせになっている。将来的には、固定の Standard / FinalStage 定義ではなく、共通列定義と自由形式列定義を合成する形に寄せられる。

### Markdown 出力

一番強く具象型に依存しているのは `FinalRankingMarkdownFileWriter`。

`CreateResultMarkdownCore` と内部のモデル生成で、`IReadOnlyList<StandardResultRow>` と `IReadOnlyList<FinalStageResultRow>` を型分岐している。

Markdown は CSV と違い、列だけでなく次の意味付けを持つ。

- 注目点
- 表の列順
- 棋士ごとの補足欄
- グループ別の見出し
- チャート対象の値

そのため、自由形式部分は単なる追加列リストでは足りない。`列` と `レポート表示部品` の両方を表せる profile が必要。

## 提案する設計方向

### 0. 内部モデルを入力・出力から独立させる

内部側は、入力データの列名や出力レポートの列名を知らない状態を目標にする。

入力側は、STSAInput、初期対局予定テキスト、大会ルール定義などを読み、内部側が扱える共通概念へ変換する。出力側は、内部結果を受け取り、CSV / Markdown 用の列や表示部品へ変換する。

### 1. 統一行モデルを追加する

新しい行モデルは、ルール名ではなく次のような構造にする。

- `SimulationResultRowCommonData CommonData`
- `IReadOnlyList<SimulationResultExtraColumn> ExtraColumns`
- `IReadOnlyDictionary<string, SimulationResultMetric> Metrics`

`ExtraColumns` は CSV や Markdown 表に出す値を保持する。
`Metrics` は、注目点やチャートなど、列表示以外にも使う意味付きの値を保持する。

### 2. 既存行型はすぐ消さず、変換口を作る

最初の実装では `StandardResultRow` / `FinalStageResultRow` をいきなり削除しない。

まずは既存の `RankingResultRowBuilder` が、従来行型と同じ内容を持つ統一行モデルを作れるようにする。出力内容が一致することを確認してから、呼び出し側を順に統一行モデルへ寄せる。

### 3. CSV writer を先に寄せる

CSV はすでに共通処理が多いので、最初の移行対象に向いている。

`GetFinalRankingCsvSpecificColumns()` を行型側の固定メソッドにするのではなく、統一行モデルの `ExtraColumns` を列定義順に書き出す形に変える。

### 4. Markdown writer は profile 化する

Markdown は次のような report profile を受け取る形に寄せる。

- 表に出す自由形式列
- 注目点に使う metric
- チャートに使う metric
- グループ別セクションの有無
- 補足欄の作り方

これにより、`Standard` / `FinalStage` という型分岐ではなく、「このレポート profile がどの表示部品を持つか」で出力を切り替える。

## 境界項目の利用域を記録する

4大境界データは、境界単位だけでなく、1項目ごとに4大域のどこで使われるかを追跡する。

目的は、内部側に不要なデータ知識が入り込んでいないかを見つけること。例えば、［シミュレーション域］では使わない項目、［大会利用者域］と［順位付け域］だけで使う項目、［出力側］だけで表示ラベルとして使う項目を分けて見えるようにする。

設計には、次のような利用表を追加する。

| 境界 | 項目 | 大会利用者域 | シミュレーション域 | 順位付け域 | 大会品質評価フロー域 | 備考 |
|---|---|---:|---:|---:|---:|---|
| Request / TournamentRule | `RuleProfileMode` | ○ | △ | △ | △ | 現状はフロー分岐に使う。将来は内部計算から切り離したい。 |
| Request / TournamentRule | `FirstPlayerWinRatePercent` | ○ | ○ | × | △ | シミュレーション入力。品質評価で条件表示に使う可能性あり。 |
| Request / PlayerList | `Players` | ○ | ○ | △ | △ | シミュレーションの主入力。順位付け・品質評価では名前や基準順位参照に使う可能性あり。 |
| Request / RankingSettings | `IsIntermediate` | ○ | △ | ○ | △ | 中間順位か最終順位かの扱い。 |
| TournamentFinalState | `MatchRecords` | × | ○ | ○ | △ | シミュレーション出力で、順位付け入力。 |
| FinalRanking | `RankRows` | △ | × | ○ | ○ | 品質評価の主入力。シミュレーション域は知らない方針。 |
| TournamentQualityReport | `Summary` | ○ | × | × | ○ | 大会利用者へ返す評価結果。 |

記号の意味は次の通り。

- `○`: その大域の主処理で使う
- `△`: 現状または表示・補助用途で参照する可能性がある
- `×`: 原則として使わない

この表は最初から完全に埋め切らず、実装調査と移行のたびに更新する。特に `△` は、残すべき参照か、入力側・出力側へ逃がすべき参照かを判断する対象にする。
## 段階計画

1. 入力側・内部側・出力側の責務境界を既存コード上で確認する。
2. 4大境界データの項目ごとに、4大域の利用表を作る。
3. 統一行モデルと自由形式列モデルを追加する。既存出力にはまだ接続しない。
4. `StandardResultRow` と `FinalStageResultRow` から統一行モデルを作る変換処理を追加する。
5. CSV writer を統一行モデル対応にする。既存 CSV と差分が出ないことを確認する。
6. Markdown writer の表部分だけを統一行モデル対応にする。
7. Markdown writer の注目点・チャート・グループ別表示を report profile へ移す。
8. `SimulationMainlineExecutionResult<StandardResultRow>` / `<FinalStageResultRow>` の分岐を、統一行モデルの実行結果へ寄せる。
9. 既存行型がアダプターだけになった段階で、削除するか互換用に残すか判断する。

## リスク

- Markdown は列だけでは表現できないため、自由形式部分を `string` 列だけにすると後で詰まる。
- `FinalStage` のグループ別表示は業務意味を持つため、共通部分へ入れず report profile 側へ置く。
- `RuleProfileMode` は入力モード選択としてまだ残る。今回の目的は、結果行と出力の分岐をルール名から切り離すこと。
- 将来 DSL 化するとき、内部側が DSL の構文やファイル形式を直接知ると再び密結合になる。DSL 解釈は入力側に置き、内部側へは解釈済みの実行計画を渡す。
- 移行途中で CSV と Markdown の列順が崩れやすいので、まず既存出力との一致確認を優先する。

## 第62ステップで実装しないこと

- `StandardResultRow` / `FinalStageResultRow` の削除
- `RuleProfileMode` の削除
- 入力ファイル仕様の変更
- Markdown テンプレートの大規模変更
- シミュレーション計算ロジックの変更

## 実装メモ

2026-05-31 時点で、既存出力を切り替えない足場として次を追加した。

- `SimulationResultFreeColumn`: ルールやレポートごとに変わる自由形式列
- `SimulationResultMetric`: 注目点やチャートで使う意味付き数値
- `GeneralSimulationResultRow`: `SimulationResultRowCommonData` と自由形式列・metric を持つ統一行モデル
- `StandardResultRow.ToGeneralResultRow()`
- `FinalStageResultRow.ToGeneralResultRow()`
- `RankingResultRowBuilder.BuildGeneralResultRows()`
- `RankingResultRowBuilder.BuildFinalStageGeneralResultRows()`

この段階では、既存 writer の呼び出し先はまだ変更していない。次は CSV writer を統一行モデルへ寄せる前に、列定義順と自由形式列キーの対応を決める。
## 確認方法

実装ステップに入ったら、少なくとも次を確認する。

- `dotnet build`
- 既存 smoke 入力での CSV 出力差分確認
- 既存 smoke 入力での Markdown 出力差分確認
- `FinalStage` のグループ分け On / Off の両方
