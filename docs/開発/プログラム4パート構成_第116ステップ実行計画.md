# プログラム4パート構成 第116ステップ実行計画

## 目的

手入力フローで最初に選ぶ「通常ルール / 本戦ルール / 大会進行フレームワーク / 空ルール」を、内部では `RuleProfileMode` ではなく `RuleProfileAttributes` として持ち回る。

利用者向けの選択 UI は互換として残す。今回の狙いは、手入力の実行・要求ファイル化・旧 fallback 実行が、分類ラベルではなく属性を受け取れる入口を持つこと。

## 実施内容

1. `ConsolePromptReaders.ReadRuleProfileAttributes(...)` を追加する。
   - 既存の `ReadRuleProfileMode(...)` は互換 UI として残す。
   - 読み取った互換ラベルはすぐ `RuleProfileAttributes.FromCompatibilityLabel(...)` へ変換する。
2. `ApplicationTournamentUser` の手入力状態を属性中心へ変更する。
   - `TournamentUserDomainResult` は `RuleProfileAttributes` を保持する。
   - 互換表示用の `RuleProfileMode` は `RuleProfileAttributes.ToCompatibilityLabel()` から導出する。
3. `ManualAnalysisRequestReader` / `ManualSimulationRequestReader` / `ManualQualityEvaluationRequestReader` の入口を属性受け取りへ変更する。
   - シミュレーションは `SimulationShape` と `UsesFinalStageGrouping` で実行先を決める。
   - 品質評価は `PairingSource=ScheduledMatches` の属性だけを受け付ける。
   - 既存の品質評価入力 reader が `RuleProfileMode` を要求する箇所では、一時的に互換ラベルへ戻す。
4. legacy の `SimulationFlowDispatcher` / `QualityEvaluationFlowDispatcher` に属性受け取り overload を追加する。
   - 既存の `RuleProfileMode` 入口は互換 wrapper として維持する。
   - `ApplicationWorkflow` は `TournamentUserDomainResult.RuleProfileAttributes` を渡す。

## 変えないこと

- 手入力の表示文言と番号選択。
- `STSAInput/4` / `STSAInput/5` の要求ファイル形式。
- `StandardSimulationRequest` / `FinalStageSimulationRequest` など既存 request 型。
- 品質評価入力 reader の内部構造。ここは後続で属性受け取りへ寄せる。

## 実施結果

- 手入力の対象ルール選択を、互換 `RuleProfileMode` ではなく `RuleProfileAttributes` として保持するよう変更した。
- `TournamentUserDomainResult` は属性を保持し、互換ラベルは導出プロパティにした。
- 手入力 request reader と `ApplicationWorkflow` 直下の legacy dispatcher 呼び出しは属性を渡す形へ変更した。
- dispatcher 側には互換 overload を残し、既存呼び出し元を壊さない形にした。
- `dotnet build`、`git diff --check`、Standard STSAInput/4 シミュレーション smoke、Standard STSAInput/5 品質評価 smoke は成功した。
- smoke 生成物は実装差分から外すため削除済み。
