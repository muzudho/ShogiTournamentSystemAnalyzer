# プログラム4パート構成 第137ステップ実行計画

## 目的

`AnalysisRequest.cs` に残っている `Standard*Request` / `FinalStage*Request` という旧仕様由来のストラテジー名を、完全に［ルールの個別プロパティ］別の要求構造へ組み直す。

ここでいう「完全に」は、単なる型名変更ではなく、実行可否や実行先選択を `Standard` / `FinalStage` という大分類 record で表さない状態を指す。`STSAInput/4` の `RuleProfileMode` は互換入力として残してよいが、parser 直後に `RuleProfileAttributes` と個別部品へ変換し、以後の内部実行では互換ラベルを使わない。

## 現状の問題

現在の `AnalysisRequest` は、`RuleProfileAttributes` を導入済みでも、具体要求型が旧大分類を背負っている。

- `StandardSimulationRequest`
- `FinalStageSimulationRequest`
- `StandardQualityEvaluationRequest`
- `DeferredStandardQualityEvaluationRequest`
- `FinalStageQualityEvaluationRequest`
- `DeferredFinalStageQualityEvaluationRequest`

その結果、次の層が型名で標準/本戦を判断している。

- `SimulationDomain.TryExecute(...)`
- `QualityEvaluationDomain.TryExecute(...)`
- `StsaInputRequestParser.ParseStepRequest(...)`
- `ManualAnalysisRequestReader`
- `StsaInputRequestWriter`
- `AnalysisStepRuleProfileAttributes`

この構造だと、`UsesFinalStageGrouping` / `UsesAdditionalApexPlacement` / `UsesBoundaryRescue` / `UsesVariableTop8` / `HasReferenceMatches` を個別に増減する要求を表しにくい。

## ゴール

1. `AnalysisStepRequest` の具象型から `Standard` / `FinalStage` という大分類名を外す。
2. シミュレーション要求は、`RuleProfileAttributes` と個別部品の組み合わせで表す。
3. 品質評価要求は、標準/本戦別 record ではなく、`TournamentQualityEvaluationRuleDefinition` と `TournamentQualityEvaluationInput` の中身で表す。
4. deferred 品質評価も標準/本戦別 record に分けず、`RuleProfileAttributes` と後続補完オプションで表す。
5. `STSAInput/5` の `RuleProfileAttributes` はそのまま正の入力仕様として維持する。
6. `STSAInput/4` の `RuleProfileMode` は互換入力/互換出力としてだけ維持する。

## 新しい要求構造案

### 共通

```csharp
internal abstract record AnalysisStepRequest;

internal sealed record SimulationStepRequest(
    RuleProfileAttributes RuleProfileAttributes,
    double FirstPlayerWinRatePercent,
    ScheduledMatchesSimulationInput? ScheduledMatchesInput,
    FinalStageGroupingRequest? FinalStageGrouping,
    AdditionalApexPlacementRequest? AdditionalApexPlacement,
    BoundaryRescueRequest? BoundaryRescue,
    TournamentFrameworkSimulationInput? TournamentFrameworkInput,
    int? SimulationCount,
    string? OutputPath) : AnalysisStepRequest;

internal sealed record QualityEvaluationStepRequest(
    RuleProfileAttributes RuleProfileAttributes,
    TournamentQualityEvaluationRuleDefinition RuleDefinition,
    TournamentQualityEvaluationInput Input,
    TournamentQualityEvaluationExecutionOptions ExecutionOptions,
    TournamentQualityEvaluationOutputOptions OutputOptions) : AnalysisStepRequest;

internal sealed record DeferredQualityEvaluationStepRequest(
    RuleProfileAttributes RuleProfileAttributes,
    TournamentQualityEvaluationExecutionOptions ExecutionOptions,
    TournamentQualityEvaluationOutputOptions OutputOptions,
    DeferredQualityEvaluationOptions DeferredOptions) : AnalysisStepRequest;
```

### シミュレーション入力部品

```csharp
internal sealed record ScheduledMatchesSimulationInput(
    TournamentRuleSetMode TournamentRuleSetMode,
    IReadOnlyList<Player> AllPlayers,
    IReadOnlyList<Player> Players,
    IReadOnlyList<Match> Matches,
    IReadOnlyList<Match> ReferenceMatches);

internal sealed record FinalStageGroupingRequest(
    FinalStageGroupingMode GroupingMode,
    IReadOnlyDictionary<string, FinalStageGroup> GroupMap,
    int ApexCount,
    int InnovCount);

internal sealed record AdditionalApexPlacementRequest(
    IReadOnlyList<Player> AdditionalApexPlayers,
    AdditionalApexPlacementMode Mode,
    int EffectiveAdditionalApexCount);

internal sealed record BoundaryRescueRequest(
    BoundaryRescueMode Mode);

internal sealed record TournamentFrameworkSimulationInput(
    string PlayersCsvPath,
    string StagesCsvPath,
    string TournamentMatchRecordsCsvPath,
    string? RuleFilePath,
    int? RandomSeed,
    TournamentRuleSetMode TournamentRuleSetMode);
```

`EmptySimulationRequest` は独立型にせず、`SimulationStepRequest.RuleProfileAttributes.SimulationShape == Empty` かつ入力部品なしで表す。必要なら `EmptySimulationInput` を追加してもよいが、出力先だけなら `OutputPath` で足りる。

### Deferred 品質評価

```csharp
internal sealed record DeferredQualityEvaluationOptions(
    VariableTop8Mode VariableTop8Mode,
    TournamentQualityEvaluationInnovExpectedRankOffsetMode InnovExpectedRankOffsetMode);
```

標準相当では `VariableTop8Mode=Off` / `InnovExpectedRankOffsetMode=Off` を入れる。どちらの値が意味を持つかは `RuleProfileAttributes` で判定する。

## 実施順

### 1. 新しい型を追加し、旧型と併存させる

- `AnalysisRequest.cs` に新しい部品 record を追加する。
- 既存の `Standard*` / `FinalStage*` 型は一旦残す。
- `AnalysisStepRuleProfileAttributes` に新型対応を追加する。
- この段階では実行経路は変えず、ビルドだけ通す。

### 2. parser から新型を作る入口を追加する

- `StsaInputRequestParser.ParseStepRequest(...)` 内に、新型生成 helper を追加する。
- まず `Simulation + ScheduledMatches` を `SimulationStepRequest` で作れるようにする。
- `UsesFinalStageGrouping=Off` の場合は `FinalStageGrouping` / `AdditionalApexPlacement` / `BoundaryRescue` を null にする。
- `UsesFinalStageGrouping=On` の場合は、既存の本戦 parser と同じ検証を行い、各部品へ詰める。
- `TournamentFramework` と `Empty` も同じ `SimulationStepRequest` へ詰める。
- 旧型生成 helper は rollback 用に残し、まだ呼び出し先を切り替えない。

### 3. SimulationDomain を新型対応にする

- `SimulationDomain.TryExecute(...)` に `SimulationStepRequest` 分岐を追加する。
- 実行先選択は record 型ではなく、`RuleProfileAttributes` と部品の有無で行う。
- `ScheduledMatchesInput` があり、`UsesFinalStageGrouping=Off` なら既存 `StandardModeSimulationContext` を組む。
- `ScheduledMatchesInput` と `FinalStageGrouping` があり、`UsesFinalStageGrouping=On` なら既存 `FinalStageModeSimulationContext` を組む。
- `TournamentFrameworkInput` があれば既存 `TournamentFrameworkModeContext` を組む。
- `SimulationShape=Empty` なら既存 `SimulationEmptyMode.Run(...)` を呼ぶ。
- 旧型分岐は一旦残す。

### 4. QualityEvaluationDomain を新型対応にする

- `QualityEvaluationStepRequest` は標準/本戦に分けず、そのまま `TournamentQualityEvaluationMainline.Run(...)` へ渡す。
- `DeferredQualityEvaluationStepRequest` は、`context.LastSimulationRequest` を新型の `SimulationStepRequest` として読む。
- `RuleProfileAttributes.UsesFinalStageGrouping` で補完内容を分ける。
- 標準相当では group / additional apex / boundary rescue なしの `TournamentQualityEvaluationRuleDefinition` を作る。
- 本戦相当では `FinalStageGrouping` / `AdditionalApexPlacement` / `BoundaryRescue` 部品から `TournamentQualityEvaluationRuleDefinition` を作る。
- 旧 deferred 型分岐は一旦残す。

### 5. 実行コンテキストを新型へ寄せる

- `AnalysisExecutionContext.LastSimulationRequest` が `AnalysisStepRequest` を持っている場合は、新型補完で型チェックできるようにする。
- 必要なら `LastSimulationStepRequest` のような新プロパティを追加し、旧型と併存させる。
- deferred 品質評価 smoke をここで必ず通す。

### 6. 手入力 reader を新型生成へ切り替える

- `ManualSimulationRequestReader` は、選択された `RuleProfileAttributes` から `SimulationStepRequest` を作る。
- `ManualQualityEvaluationRequestReader` は、`StandardQualityEvaluationRequest` / `FinalStageQualityEvaluationRequest` を作らず、常に `QualityEvaluationStepRequest` を作る。
- 表示文言の「標準」「本戦」は利用者向けラベルとして残してよいが、型名や内部分岐には使わない。

### 7. STSAInput writer を新型へ寄せる

- `StsaInputRequestWriter` は `RuleProfileAttributes` と部品の有無から出力する。
- `STSAInput/5` は `RuleProfileAttributes` / `Step.*.RuleProfileAttributes` をそのまま出力する。
- `STSAInput/4` は互換出力として `ToCompatibilityLabel()` を使う。
- 旧型出力分岐は parser / domain 切り替え完了後に削除する。

### 8. parser の本線を新型へ切り替える

- `StsaInputRequestParser.ParseStepRequest(...)` が返す型を新型に切り替える。
- `EnsureStandardSimulationCountIfNeeded` / `EnsureFinalStageSimulationCountIfNeeded` のような大分類名 helper を、`EnsureSimulationCountIfNeeded(SimulationStepRequest request)` へ統合する。
- 品質評価の `EnsureStandard...` / `EnsureFinalStage...` も `EnsureQualityEvaluationSimulationCountIfNeeded(QualityEvaluationStepRequest request)` へ統合する。

### 9. 旧型を削除する

以下の型を削除する。

- `StandardSimulationRequest`
- `FinalStageSimulationRequest`
- `EmptySimulationRequest`
- `StandardQualityEvaluationRequest`
- `DeferredStandardQualityEvaluationRequest`
- `FinalStageQualityEvaluationRequest`
- `DeferredFinalStageQualityEvaluationRequest`

`TournamentFrameworkSimulationRequest` も `SimulationStepRequest + TournamentFrameworkSimulationInput` に吸収できた段階で削除する。

### 10. TODO コメントと docs を更新する

- `AnalysisRequest.cs` の旧仕様 TODO を削除する。
- `docs/続きはここから.md` に実施結果と残課題を書く。
- 必要なら `STSAInput/5 仕様` に、内部では互換ラベルではなく属性と部品で処理する旨を追記する。

## 検証計画

各段階で最低限、次を通す。

- `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj`
- `git diff --check`

切り替え段階では次の smoke を通す。

- Standard 相当の STSAInput/5 シミュレーション smoke
- FinalStage 相当の STSAInput/5 シミュレーション smoke
- Standard 相当の STSAInput/5 品質評価 smoke
- FinalStage 相当の STSAInput/5 品質評価 smoke
- `Simulation,QualityEvaluation` 複数ステップ STSAInput/5 smoke
- `--smoke-request-writer-roundtrip --format STSAInput/5`
- `--smoke-request-parse-fails` の不整合属性 smoke

## リスク

- `SimulationDomain` と `QualityEvaluationDomain` が旧具象型に強く依存しているため、一括削除すると deferred 品質評価が壊れやすい。
- `StsaInputRequestWriter` は型 switch が多く、parser より遅れて切り替えると round-trip smoke が落ちやすい。
- `RuleProfileAttributes.TryValidate(...)` は現在、代表プロファイル前提の検証になっている。個別プロパティの自由度を本当に上げるなら、検証規則を「代表4パターン」から「部品同士の整合性」へ緩める必要がある。
- `TournamentQualityEvaluationRuleDefinition` 側はすでに個別プロパティに近いが、`TournamentQualityEvaluationOutputProfile.Standard/FinalStage` のような出力 profile 名は残る。これは出力形式の互換ラベルとして扱うか、別ステップで改名する。

## 着手時の推奨スライス

最初の実装スライスは、旧型削除まで狙わない。

1. `SimulationStepRequest` と部品 record を追加する。
2. `SimulationDomain` に新型分岐を追加する。
3. parser の `Simulation + ScheduledMatches` だけ新型へ切り替える。
4. Standard 相当シミュレーション smoke と writer round-trip smoke を通す。

このスライスで「大分類型ではなく、属性と部品で実行できる」最小の縦線を作る。その後に本戦相当、TournamentFramework、Empty、品質評価、deferred の順で広げる。
