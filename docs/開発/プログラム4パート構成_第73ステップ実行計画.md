# プログラム4パート構成 第73ステップ実行計画

## 目的

`ApplicationWorkflow` に残っている `RequestBoundary` と `AnalysisRequest` の二重管理を、設計意図が分かる形へ整理する。

`AnalysisRequest` は STSAInput/4 をパースした新しい本命ルートの要求オブジェクトとして育てる。`RequestBoundary` は旧対話ルート用の薄い境界であり、`Application` 層の `AnalysisRequest` を持たせて太らせない。

## 判断

- `AnalysisRequest` は `Application.RequestParsing` の型なので、`Domain.Request.RequestBoundary` へ持たせると Domain から Application への逆依存になりやすい。
- 旧ルートに必要なのは `AnalysisFlowSelection` と `RuleProfileMode` だけなので、`ApplicationWorkflow` のローカル状態として持てばよい。
- `RunAnalysisDomain` は `RequestBoundary` 全体を受け取らず、旧ルート実行に必要な選択値と、新ルート用の `AnalysisRequest?` だけを受け取る。

## 今回の実装スライス

1. `ApplicationWorkflow.Run` から `RequestBoundary` 生成を外す。
2. `analysisFlowSelection` / `ruleProfileMode` を `ApplicationWorkflow` のローカル状態として持つ。
3. `RunTournamentUserDomain` は、手入力・legacy fallback・STSAInput/4 直通ルートの結果をこの2値と `AnalysisRequest?` に反映する。
4. `RunAnalysisDomain` は `RequestBoundary` ではなく、`AnalysisFlowSelection` / `RuleProfileMode` / `AnalysisRequest?` を受け取る。

## 実装メモ

- `RunTournamentUserDomain` の `ref` 引数群は、`TournamentUserDomainResult` へまとめた。
- `Run` は大会利用者域の結果オブジェクトを受け取り、分析実行と要求ファイル書き出しへ渡す。
- コード上未使用だった `RequestBoundary` と、そのためだけに残っていた空の `PlayerListBoundary` / `RankingSettingsBoundary` / `TournamentRuleBoundary` を削除した。
- 手入力のうち `Simulation + TournamentFramework` は、`SimulationModeInputReaders.ReadTournamentFrameworkModeContext()` の結果から `TournamentFrameworkSimulationRequest` を組み立て、`AnalysisRequestDispatcher` で実行する入口を追加した。
