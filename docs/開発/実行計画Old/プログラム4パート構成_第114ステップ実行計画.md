# プログラム4パート構成 第114ステップ実行計画

## 目的

`RuleProfileMode.Standard` / `RuleProfileMode.FinalStage` のような大分類を、将来的にシステム内部の分岐軸から廃止する。

ルールの違いは、分類名ではなく、ルールプロファイルの属性で表現する。既存入力ファイルと smoke を壊さないため、分類名は互換入力として残しながら、内部実行の判断材料を段階的に属性へ移す。

## 現状の障害

- `RuleProfileMode` は `Domain/TournamentQualityEvaluator/TournamentQualityEvaluatorEnums.cs` に enum としてあり、`Standard` / `FinalStage` / `TournamentFramework` / `Empty` を持つ。
- `AnalysisRequest` がトップレベルに `RuleProfileMode` を持つため、要求全体が単一の大分類に縛られている。
- `STSAInput/4` は単一ステップでも複数ステップでも `RuleProfileMode` を必須メタ情報として読む。
- `StsaInput4RequestWriter` は `AnalysisStepRequest` の具象型から `RuleProfileMode` を逆算して書き出す。
- 手入力は `ConsolePromptReaders.ReadRuleProfileMode(...)` で大分類を先に選ばせる。
- legacy 入口の `SimulationFlowDispatcher` / `QualityEvaluationFlowDispatcher` は `RuleProfileMode` switch で実行先を決める。
- 最終順位表 writer と品質評価出力オプションが、出力スキーマや表示用途の判定に `RuleProfileMode` を使っている。

これらを一度に消すと、入力互換、手入力、出力ファイル名、品質評価 deferred 実行が同時に崩れる。先に属性モデルを追加し、既存分類から属性へ変換する薄い互換層を置く必要がある。

## 方針

大分類名を「入力互換ラベル」に格下げし、内部実行は `RuleProfileAttributes` で判断する。

最初に持たせる属性候補:

- `SimulationShape`: `ScheduledMatches` / `FinalStageGrouped` / `TournamentFramework` / `Empty`
- `UsesFinalStageGrouping`: Apex / Innov グループを使うか
- `UsesAdditionalApexPlacement`: 本戦不出場 Apex の扱いを使うか
- `UsesBoundaryRescue`: 境界救済戦を使うか
- `UsesVariableTop8`: 可変定員 8 ルールを使うか
- `RankingRuleSetMode`: `Neutral` / `Twill` / `TwillCommonOpponentWeighted`
- `HasReferenceMatches`: 比較用対局表を持つか
- `PairingSource`: 対局表固定 / 大会進行フレームワーク / なし

名称は実装時に既存の `TournamentQualityEvaluationRuleDefinition` と重ならないよう調整する。

## 実施内容

1. `RuleProfileAttributes` を追加する。
   - まずは既存 `RuleProfileMode` から属性を作る adapter を用意する。
   - `Standard` は `ScheduledMatches + UsesFinalStageGrouping=false` として表現する。
   - `FinalStage` は `ScheduledMatches + UsesFinalStageGrouping=true` として表現する。
   - `TournamentFramework` と `Empty` も属性で表現する。
2. `AnalysisStepRequest` から属性を取得できる入口を追加する。
   - トップレベル `AnalysisRequest.RuleProfileMode` への依存を減らす。
   - 複数ステップ要求では、各 step の属性を正とする。
3. `SimulationFlowDispatcher` / `QualityEvaluationFlowDispatcher` の legacy 分岐を薄くする。
   - `RuleProfileMode` switch は互換 adapter 呼び出しに寄せる。
   - 実行先の判断は、可能な範囲で `AnalysisStepRequest` または属性へ移す。
4. `StsaInput4RequestParser` は当面 `RuleProfileMode` を読み続ける。
   - 読んだ直後に属性へ変換し、内部要求に属性を添える。
   - 新しい属性指定形式を追加する場合は `STSAInput/5` か `STSAInput/4` 拡張として別ステップで判断する。
5. `StsaInput4RequestWriter` は互換出力として `RuleProfileMode` を維持する。
   - ただし内部的には step 属性から互換ラベルを導出する。
   - 属性だけで表現できない古い分類依存が残っていれば、その箇所を明示的に洗い出す。
6. 最終順位表 writer と品質評価 output options の用途を分離する。
   - 出力スキーマ選択に必要なのは「標準表か本戦表か」であり、分類名そのものではない。
   - ここを `FinalRankingTableProfile` のような出力専用設定へ切り出す。
7. すべての smoke が通った後、`RuleProfileMode.Standard` / `FinalStage` を新規コードの分岐軸として使わないルールを設ける。

## 変えないこと

- 既存の `STSAInput/4` 要求ファイル互換。
- 既存の手入力 UI 文言。
- `StandardSimulationRequest` / `FinalStageSimulationRequest` の型名。第114ステップでは型名変更まで踏み込まない。
- 計算ロジック、順位計算、品質評価指標。
- 出力 CSV / Markdown の内容。

## 実施結果

- `RuleProfileAttributes` を追加し、既存 `RuleProfileMode` から属性へ変換する adapter と互換ラベル復元を用意した。
- `AnalysisStepRuleProfileAttributes` を追加し、`AnalysisStepRequest` の具象型から属性を導出できる入口を作った。
- legacy の `SimulationFlowDispatcher` / `QualityEvaluationFlowDispatcher` と `SimulationScenarioFactory` は、`RuleProfileMode` switch ではなく属性の `SimulationShape` / `PairingSource` / `UsesFinalStageGrouping` を見る形へ寄せた。
- `StsaInput4RequestWriter` は、要求全体の `RuleProfileMode` ではなく step 属性から互換 `RuleProfileMode` を書き出すようにした。
- 最終順位 writer は `FinalRankingTableProfile` を追加し、出力スキーマ選択を `RuleProfileMode` から切り離した。
- 検証として `dotnet build`、`git diff --check`、Standard / FinalStage / 複数ステップ smoke が成功した。
## 次ステップ候補

- 第117ステップ: `STSAInput/5` として `RuleProfileMode` ではなく属性セクションを持つ入力形式を追加する。
