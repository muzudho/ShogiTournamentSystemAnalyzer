# プログラム4パート構成 第132ステップ実行計画

## 目的

旧入力互換層の `RuleProfileMode` 利用を、互換ラベル読み取り直後に `RuleProfileAttributes` へ寄せる。

第131ステップまでで通常実行・要求ファイル writer・出力オプション側は属性中心へ寄った。今回は `STSAInput/4` 互換入力の読み取りは維持しつつ、直通 parser と legacy converter が互換 enum を持ち回る範囲を狭める。

## 実施内容

1. `StsaInputValueParser` に `ParseRuleProfileAttributesFromCompatibilityLabel(...)` を追加する。
2. `StsaInputRequestParser` の `STSAInput/4` 読み取りを、互換ラベル parse 後すぐ `RuleProfileAttributes` へ変換する入口へ差し替える。
3. `StsaInputLegacyConverter` の分岐と prompt prefix 生成を `RuleProfileAttributes` 判定へ寄せる。

## 変えないこと

- `STSAInput/4` の `RuleProfileMode` 入力互換。
- `StsaInputRequestWriter` が `STSAInput/4` 出力時に `RuleProfileMode` を書く動作。
- `RuleProfileMode` enum 自体。互換ラベルの境界型として残す。
- 計算ロジック、入力 CSV、出力 CSV / Markdown の内容。

## 実施結果

- `StsaInputValueParser.ParseRuleProfileAttributesFromCompatibilityLabel(...)` を追加し、互換ラベルから属性へ変換する入口を要求ファイル値 parser に集約した。
- `StsaInputRequestParser` は `STSAInput/4` の `RuleProfileMode` を読み取った直後に `RuleProfileAttributes` へ変換するようになった。
- `StsaInputLegacyConverter` は `RuleProfileAttributes` で標準/本戦/大会進行フレームワーク/空ルール相当を判定するようになった。
