# プログラム4パート構成 第141ステップ実行計画

## 目的

`AnalysisFlowSelection` の `Steps` 配列をやめ、アプリケーション直下の処理域に対応した3つのプロパティへ置き換える。

現状の `AnalysisFlowSteps=Simulation,QualityEvaluation` は、要求ファイル上の互換表記としては有効だが、内部モデルの `Steps` 配列が「要求ファイルのステップ順」と「アプリケーション直下の処理域」を混ぜて見せている。

第141ステップでは、外部フォーマットの `AnalysisFlowSteps` は維持しつつ、内部の `AnalysisFlowSelection` を次の3大域で表す。

```csharp
RunsSimulationDomain
RunsFinalRankingDomain
RunsQualityEvaluationDomain
```

［最終順位付け域］は現状まだ独立要求ではなく、主に［シミュレーション域］の中から呼ばれている。そのため、このステップでは「最終順位付け処理の完全分離」までは行わず、内部フロー選択モデル上の位置を先に作る。

## 方針

- `AnalysisFlowSelection.Steps` を削除する。
- `AnalysisFlowSelection` は3大域の実行可否プロパティを持つ。
- `AnalysisFlowSteps` のパースと出力は、互換入出力の境界でのみ扱う。
- `AnalysisRequest.StepRequests` は、このステップでは残す。
  - 要求ファイルから読んだ具体実行要求の順序付きリストとしてまだ使う。
  - 将来、`SimulationDomainRequest` / `FinalRankingDomainRequest` / `QualityEvaluationDomainRequest` のような3プロパティ構造へ移す候補にする。
- `FinalRanking` は、現状の実行仕様に合わせて `Simulation` が指定されたときに `RunsFinalRankingDomain` も `true` にする。

## 新しい内部モデル案

```csharp
internal sealed class AnalysisFlowSelection
{
    internal AnalysisFlowSelection(
        bool runsSimulationDomain,
        bool runsFinalRankingDomain,
        bool runsQualityEvaluationDomain)
    {
        if (!runsSimulationDomain && !runsFinalRankingDomain && !runsQualityEvaluationDomain)
        {
            throw new ArgumentException("分析フローが空です。");
        }

        RunsSimulationDomain = runsSimulationDomain;
        RunsFinalRankingDomain = runsFinalRankingDomain;
        RunsQualityEvaluationDomain = runsQualityEvaluationDomain;
    }

    internal bool RunsSimulationDomain { get; }

    internal bool RunsFinalRankingDomain { get; }

    internal bool RunsQualityEvaluationDomain { get; }
}
```

互換のため、当面は旧名 wrapper も残す候補にする。

```csharp
internal bool RunsSimulation => RunsSimulationDomain;
internal bool RunsQualityEvaluation => RunsQualityEvaluationDomain;
```

ただし `Steps` 互換 property は残さない。残すと今回の目的が曖昧になるため。

## 実施内容

1. `AnalysisFlowSelection` を3大域プロパティ構造へ変更する。
   - `Steps` property を削除する。
   - `RunsSimulationDomain` / `RunsFinalRankingDomain` / `RunsQualityEvaluationDomain` を追加する。
   - 既存呼び出し互換のため `FromSingle(...)` / `FromFlags(...)` は残す。
   - `FromSingle(Simulation)` は `SimulationDomain=true` / `FinalRankingDomain=true` / `QualityEvaluationDomain=false` にする。
   - `FromSingle(QualityEvaluation)` は `SimulationDomain=false` / `FinalRankingDomain=false` / `QualityEvaluationDomain=true` にする。
   - `FromFlags(runsSimulation, runsQualityEvaluation)` は `runsSimulation` のとき `RunsFinalRankingDomain` も `true` にする。
2. `AnalysisFlowSteps` parser を境界処理へ寄せる。
   - `ParseAnalysisFlowSteps(...)` は `AnalysisFlowSelection` を返し続ける。
   - 内部で一時的な `AnalysisFlowMode[]` を作ってよいが、返却後に配列を保持しない。
   - 複数ステップ要求の parser で必要な順序は、別 helper でローカルに読み取る。
3. `StsaInputRequestParser` の単一/複数ステップ判定を `Steps` 依存から外す。
   - `ReadFlowSelection(...)` とは別に、要求ファイルの `AnalysisFlowSteps` から `AnalysisFlowMode[]` を読む入口を追加する。
   - 単一ステップ判定はローカル配列の `Length` で行う。
   - `ParseStepRequest(...)` にはローカルの `AnalysisFlowMode` を渡す。
   - `AnalysisRequest` へは新しい `AnalysisFlowSelection` と既存 `StepRequests` を渡す。
4. `StsaInputRequestWriter` の `AnalysisFlowSteps` 出力を `StepRequests` から生成する。
   - `request.FlowSelection.ToRequestFileValue()` 依存をやめる。
   - `SimulationStepRequest` は `Simulation`、`QualityEvaluationStepRequest` / `DeferredQualityEvaluationStepRequest` は `QualityEvaluation` として出す。
   - これにより `FlowSelection` は実行域選択、`StepRequests` は要求ファイル再現用の具体要求として役割を分ける。
5. `ProgramConsoleGuide` の主線表示を3大域プロパティへ合わせる。
   - `Steps.Select(...)` をやめる。
   - `RunsSimulationDomain` / `RunsFinalRankingDomain` / `RunsQualityEvaluationDomain` から表示要素を組み立てる。
   - 表示例は `SimulationDomain -> FinalRankingDomain -> QualityEvaluationDomain` のように3大域が見える形にする。
6. `ManualAnalysisRequestReader` の `Steps[0]` 依存を削除または入口ごと削る。
   - 現在の手入力フローでは、シミュレーション域と品質評価域から個別 reader を呼ぶ形が主経路。
   - 未使用であれば `TryRead(...)` の削除候補にする。
   - 残す場合は `AnalysisFlowMode` を明示引数にする。
7. `ApplicationWorkflow` の手入力保存処理を更新する。
   - 実行済み `StepRequests` から `AnalysisFlowSelection` を作る helper を追加する。
   - helper はシミュレーション request がある場合に `RunsFinalRankingDomain=true` とする。
   - `GetAnalysisFlowMode(...)` が不要になれば writer 側 helper へ集約する。
8. legacy converter の `Steps` 依存を外す。
   - 複数ステップ未対応判定は、要求ファイルから読んだローカル `AnalysisFlowMode[]` の長さで行う。
   - 疑似コンソール入力へ流す互換処理では、3大域プロパティではなく `RunsSimulationDomain` / `RunsQualityEvaluationDomain` を使う。
9. ドキュメントの TODO を更新する。
   - `docs/設計ラフ/【むずでょ原文】[フローチャート-01]アプリケーション４パート構成.md` の `Steps` TODO を、第141ステップで解消した内容へ置き換える。
   - `docs/続きはここから.md` に残る `Steps` 説明があれば、要求ファイル境界の `AnalysisFlowSteps` と内部3大域プロパティの区別へ直す。

## 変えないこと

- 要求ファイルの `AnalysisFlowSteps=Simulation,QualityEvaluation` 表記。
- `SimulationStep` / `QualityEvaluationStep` の section 名。
- `AnalysisRequest.StepRequests` のリスト構造。
- `SimulationStepRequest` / `QualityEvaluationStepRequest` / `DeferredQualityEvaluationStepRequest` の具体 request 型。
- シミュレーション、最終順位付け、品質評価の計算ロジック。
- CSV / Markdown の入出力仕様。
- ［最終順位付け域］の処理をシミュレーション域から完全分離すること。

## 注意点

- `AnalysisFlowSelection` に `Steps` を残すと、利用側がまた配列へ戻るため削除する。
- `AnalysisFlowSteps` という外部表記は残すが、内部モデル名としての `Steps` は使わない。
- `FinalRanking` は要求ファイル上の独立 step ではないため、`AnalysisFlowSteps=FinalRanking` はこのステップでは追加しない。
- 複数ステップ要求の実行順は、当面 `AnalysisRequest.StepRequests` が保持する。
- `AnalysisRequestDispatcher` はこのステップでは `StepRequests` を順に実行するままでよい。
