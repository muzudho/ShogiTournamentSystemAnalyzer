# プログラム4パート構成 第16ステップ実行計画

## 目的

第15ステップで `InputSourceConfiguration` の要求ファイル経路を小さなメソッドへ切り出した。
第16ステップでは、`--request-file` 引数読み取りの結果をモデル化し、引数読み取りと表示・フォールバック判断を分離する。

## 現状

`RequestFileArgumentReader.TryGetInputFilePath` は、`--request-file` の指定ミスを catch し、エラーメッセージを表示したうえで `null` を返している。
このため、呼び出し側から見ると「入力ファイル未指定」と「指定エラー」が同じ `null` に見える。

挙動は変えず、読み取り結果だけを明示する。

## 今回やること

1. `RequestFileArgumentReadResult` を追加する。
2. `RequestFileArgumentReader.Read(args)` が結果モデルを返すようにする。
3. `RequestFileArgumentReader` はエラー表示をしない。
4. `InputSourceConfiguration` がエラー表示と手動入力への切り替えを行う。
5. 既存のフォールバック挙動は変えない。

## 今回やらないこと

- 複数 `--request-file` 指定時の検証追加
- 引数 parser の汎用化
- エラー文言の変更

## 完了条件

- `--request-file` 未指定と指定エラーが結果モデル上で区別できる。
- `dotnet build` が成功する。
- `--request-file` 経由の空ルール smoke 実行が成功する。
