# プログラム4パート構成 第94ステップ実行計画

## 目的

`分析` の内側を、次の3ドメインへ直列化する準備を進める。

```text
分析
  -> シミュレーション域
  -> 最終順位付け域
  -> 大会品質評価域
```

`ApplicationWorkflow.Run` 直下は、これまでどおり `大会利用者域` と `分析` の大きな入口に保つ。

## 現状

- `SimulationDomain` は `AnalysisStepRequest` からシミュレーション系 request を受け取る入口になっている。
- `QualityEvaluationDomain` は品質評価系 request を受け取る入口になっている。
- `FinalRankingDomain` は最終順位表の出力パス解決と CSV / Markdown 書き出しの入口になっている。
- ただし `StandardSimulationMainline` / `FinalStageSimulationMainline` は、シミュレーション実行、最終順位行の生成、結果表示、最終順位表出力を同じ mainline 内で続けている。

## 障害になりそうな部分

一番の障害は、シミュレーション mainline の公開入口が `void Run(...)` で、外側の `SimulationDomain` が `SimulationResult` / `FinalRankingResult` を受け取れないこと。

この状態だと、後から `Analysis` を次のように並べ替えにくい。

```text
SimulationDomain
  -> SimulationResult
FinalRankingDomain
  -> FinalRankingResult
QualityEvaluationDomain
  -> TournamentQualityReport
```

## 今回の小さな先行作業

挙動を変えずに、`AbstractSimulationMainline.Run(...)` と派生 mainline の request 入口が `SimulationMainlineResult` を返せるようにする。

この段階では、次のものはまだ変えない。

- 結果表示の位置
- CSV / Markdown 出力の位置
- `SimulationDomain` の戻り値
- `AnalysisRequestDispatcher` の実行順
- STSAInput/4 の仕様

## 次の実装候補

1. `SimulationDomain.TryExecute(...)` の内部で mainline result を受け取る。
2. `SimulationDomain` に `Execute(...)` 系の戻り値つき入口を追加し、`TryExecute(...)` は互換 wrapper にする。
3. `FinalRankingDomain` に、順位行生成の入口を寄せる。
4. `StandardSimulationMainline` / `FinalStageSimulationMainline` から最終順位表出力を外へ出す。
5. `AnalysisRequestDispatcher` を `AnalysisDomain` 風の直列実行へ寄せる。
