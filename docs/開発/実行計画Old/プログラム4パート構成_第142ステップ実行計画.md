# プログラム4パート構成 第142ステップ実行計画

## 目的

［最終順位付け域］の実処理を、［シミュレーション域］から独立させる。

現状は `FinalRankingDomain` 自体は存在し、順位行生成や CSV / Markdown 出力も集約されている。ただし実行タイミングは `SimulationDomain` や旧手入力シナリオの中に残っており、アプリケーション直下の `ExecuteFinalRankingDomain()` は空の位置決めだけになっている。

第142ステップでは、標準シミュレーションと本戦シミュレーションについて、シミュレーション実行後の最終順位付け出力を［最終順位付け域］から実行する形へ移す。

## 現状

- `ApplicationWorkflow.ExecuteFinalRankingDomain()` は空。
- `AnalysisFlowSelection` は `RunsSimulationDomain` / `RunsFinalRankingDomain` / `RunsQualityEvaluationDomain` を持つ。
- `SimulationDomain.ExecuteStandardSimulation(...)` と `ExecuteFinalStageSimulation(...)` が、シミュレーション直後に `FinalRankingDomain.WriteStandardSimulationOutputs(...)` / `WriteFinalStageSimulationOutputs(...)` を呼んでいる。
- `StandardSimulationScenario` / `FinalStageSimulationScenario` の旧手入力 fallback 側も、実行計画の中で `FinalRankingDomain.Write...` を呼んでいる。
- `SimulationDomainResult` は `SimulationResult` と `FinalRankingResult` を持つが、最終順位付け出力に必要な補助情報はまだ持っていない。

## 方針

- シミュレーション域は、大会最終状態と最終順位付けに必要な材料を結果として返すだけにする。
- 最終順位付け域は、シミュレーション域の結果を受け取り、標準 / 本戦に応じた CSV / Markdown 出力を行う。
- `AnalysisExecutionContext` を、直列実行の境界データ置き場として育てる。
- まずは現行仕様の `Simulation` 指定時に暗黙で `FinalRankingDomain` も実行する形を維持する。
- 要求ファイル上の `AnalysisFlowSteps=FinalRanking` 独立指定は、このステップでは追加しない。
- `TournamentFramework` と `Empty` は戻り値が別構造または出力仕様が違うため、このステップでは最小限の現状維持にする。必要なら次ステップで専用結果型を作る。

## 実施内容

1. 最終順位付け実行用の入力データを追加する。
   - 候補名: `FinalRankingDomainInput`。
   - 標準用に必要な値は `CalculationResult TournamentFinalState`、`double FirstPlayerWinRatePercent`、`FinalRankingResult FinalRankingResult`、`string? OutputPath`。
   - 本戦用に必要な値は、標準用の値に加えて `IReadOnlyList<Player> Players`、`IReadOnlyList<Match> ReferenceMatches`、`bool WriteReferenceMatchesForMarkdown`、標準 / 本戦を表す種別。
2. `SimulationDomainResult` に最終順位付け域へ渡す値を追加する。
   - `SimulationResult` は大会最終状態として維持する。
   - `FinalRankingResult` 単体では出力に足りないため、上記の最終順位付け入力を持たせる。
   - 既存の品質評価 deferred で使う `SimulationResult` / `FinalRankingResult` の互換は壊さない。
3. `SimulationDomain` から最終順位付け出力呼び出しを外す。
   - `ExecuteStandardSimulation(...)` は mainline 実行後、`FinalRankingDomain.WriteStandardSimulationOutputs(...)` を呼ばずに `SimulationDomainResult` を返す。
   - `ExecuteFinalStageSimulation(...)` も同様に、出力に必要な本戦補助情報を result に詰めて返す。
   - `TournamentFramework` / `Empty` は戻り値 `null` のまま、既存処理を維持する。
4. `AnalysisExecutionContext` に最終順位付け待ちデータを持たせる。
   - `SetSimulationResult(...)` の中、または別メソッドで `PendingFinalRanking` を設定する。
   - 最終順位付け完了後に必要ならクリアする。
5. `FinalRankingRequestDispatcher` を追加する。
   - `AnalysisExecutionContext` の `PendingFinalRanking` を見て、`FinalRankingDomain` へ委譲する。
   - 入力がない場合は false を返すか、呼び出し位置に応じて何もしない。
6. `FinalRankingDomain` に実行入口を追加する。
   - 候補: `TryExecute(FinalRankingDomainInput input)`。
   - 標準 / 本戦の種別に応じて、既存の `WriteStandardSimulationOutputs(...)` / `WriteFinalStageSimulationOutputs(...)` を呼ぶ。
   - 既存 writer helper は活かす。
7. `AnalysisRequestDispatcher` の直列実行を更新する。
   - `SimulationRequestDispatcher.TryExecute(...)` 成功後、context にシミュレーション結果を置く。
   - `request.FlowSelection.RunsFinalRankingDomain` が true なら、`FinalRankingRequestDispatcher.TryExecute(context)` を呼ぶ。
   - その後に品質評価 step が来た場合、context の `LastSimulationResult` は従来どおり参照できる。
8. `ApplicationWorkflow` の手入力 mainline を更新する。
   - `ExecuteSimulationDomain(...)` はシミュレーション結果を `manualExecutionState.Context` に入れるだけにする。
   - `ExecuteFinalRankingDomain(...)` に `TournamentUserDomainResult` と `ManualAnalysisExecutionState?` を渡せるようにし、手入力時は context の pending 最終順位付けを実行する。
   - legacy fallback 側は `SimulationFlowDispatcher` が void なので、まずは旧シナリオ側を別途整理する。
9. 旧手入力 fallback 側を整理する。
   - `SimulationExecutionPlan` を `Action Execute` から、必要なら結果を返せる形にする。
   - または第142ステップでは新ルートを優先し、`StandardSimulationScenario` / `FinalStageSimulationScenario` の `FinalRankingDomain.Write...` は既知の残件として残す。
   - ただし「最終順位付け域の実処理を独立」の完了条件に含めるなら、ここも `FinalRankingRequestDispatcher` 経由へ寄せる。
10. ドキュメントを更新する。
    - `docs/続きはここから.md` に第142ステップの完了内容と検証結果を追記する。
    - `docs/設計ラフ/【むずでょ原文】[フローチャート-01]アプリケーション４パート構成.md` の［最終順位付け域］説明を、空の位置決めから実処理ありへ更新する。

## 推奨する段階分け

### 第142-Aステップ

新ルートの `AnalysisRequestDispatcher` / `SimulationDomain` / `AnalysisExecutionContext` / `FinalRankingRequestDispatcher` を整理する。

対象:

- STSAInput/5 要求ファイル
- STSAInput/4 直通要求ファイル
- 手入力から `SimulationRequestDispatcher.TryExecute(...)` を呼んでいる現行経路

この段階で、通常の要求ファイル実行では `SimulationDomain` から `FinalRankingDomain.Write...` 呼び出しが消える。

### 第142-Bステップ

旧手入力 fallback の `SimulationScenarioRunner` / `SimulationExecutionPlan` / `StandardSimulationScenario` / `FinalStageSimulationScenario` を結果返却型へ寄せる。

この段階で、シナリオ内の `FinalRankingDomain.Write...` 呼び出しも消える。

### 第142-Cステップ

`TournamentFramework` と `Empty` の扱いを決める。

- `TournamentFramework` は aggregate 最終順位、representative 順位表、代表大会最終状態をまとめて出力しているため、専用の `TournamentFrameworkFinalRankingDomainInput` が必要。
- `Empty` は最終順位表出力だけだが、`SimulationDomainResult` を返していないため、空ルール用の結果型を作るか、現状維持にするかを決める。

## 変えないこと

- 要求ファイル上の `AnalysisFlowSteps=Simulation,QualityEvaluation` 表記。
- `FinalRanking` を外部 step として追加すること。
- シミュレーション計算ロジック。
- 順位行の計算ロジック。
- CSV / Markdown のファイル形式。
- 品質評価 deferred の基本仕様。

## 注意点

- `FinalRankingResult` は品質評価にも使うため、最終順位付け域で出力済みかどうかとは分けて扱う。
- 最終順位付け出力に必要な `OutputPath` は、元の `SimulationStepRequest` から引き継ぐ。
- 本戦 Markdown 用の `writeReferenceMatchesForMarkdown` は `SimulationMainlineResult.Presentation` に依存するため、シミュレーション結果から落とさない。
- `TournamentFramework` は `FinalRankingDomain` にかなり寄っているが、呼び出し主導はまだシミュレーション側なので、標準 / 本戦と同じ変更で無理に混ぜない。
- `AnalysisRequestDispatcher` でシミュレーション直後に最終順位付けを実行すると、外部 step の順序は変えずに内部3大域の順序へ近づけられる。

## 完了条件

- 新ルートの標準 / 本戦シミュレーションで、`SimulationDomain` が最終順位表を直接出力しない。
- `AnalysisExecutionContext` に置かれたシミュレーション結果を、［最終順位付け域］が読み取って出力する。
- `ApplicationWorkflow.ExecuteFinalRankingDomain()` が、少なくとも手入力の現行結果に対して実処理を持つ。
- 既存の代表 STSAInput/5 標準シミュレーション、本戦シミュレーション、複数ステップ `Simulation,QualityEvaluation` が動く。
- 品質評価 deferred が、最終順位付け域分離後も直前シミュレーション結果を参照できる。

## 次ステップ候補

- 要求ファイル上に `FinalRanking` 独立 step を追加するか検討する。
- `TournamentFramework` の aggregate / representative 出力を最終順位付け域の独立実行入力へ移す。
