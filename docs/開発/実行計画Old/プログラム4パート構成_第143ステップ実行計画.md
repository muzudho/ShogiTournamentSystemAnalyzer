# プログラム4パート構成 第143ステップ実行計画

## 目的

`AnalysisRequest` の旧 `StepRequests` リスト依存を廃止し、内部要求モデルを［シミュレーション域］［最終順位付け域］［大会品質評価域］の3プロパティ構造へ寄せる。

第141ステップで `AnalysisFlowSelection` は `RunsSimulationDomain` / `RunsFinalRankingDomain` / `RunsQualityEvaluationDomain` の3大域に整理済み。一方で `AnalysisRequest` はまだ `IReadOnlyList<AnalysisStepRequest> StepRequests` を持っており、要求ファイル parser / writer / dispatcher / 手入力 request 書き出しがこのリストを前提にしている。

## 方針

- `AnalysisRequest` は3大域プロパティを持つ。
  - `SimulationDomainRequest?`
  - `FinalRankingDomainRequest?`
  - `QualityEvaluationDomainRequest?`
- 既存の具体要求型 `SimulationStepRequest` / `QualityEvaluationStepRequest` / `DeferredQualityEvaluationStepRequest` は、実行入力として維持する。
- 外部要求ファイルの `AnalysisFlowSteps=Simulation,QualityEvaluation` 表記は維持する。
- `FinalRanking` は外部 step として追加しない。シミュレーションを実行する要求では内部的に最終順位付け域も実行対象として表す。
- writer の出力形式は現行の STSAInput/4 / STSAInput/5 と同じにする。

## 実施内容

1. `AnalysisRequest` の record 定義を3大域プロパティへ変更する。
2. `AnalysisRequest` から実行順の step 一覧を得る helper を追加する。
   - writer と dispatcher はこの helper 経由で `Simulation` -> `QualityEvaluation` の順に扱う。
   - `StepRequests` プロパティは追加しない。
3. `StsaInputRequestParser` の組み立てを新構造へ変更する。
   - 単一 step は該当大域プロパティだけを持つ request にする。
   - 複数 step は simulation / quality evaluation をそれぞれ該当大域へ格納する。
4. `StsaInputRequestWriter` を新構造へ追従する。
   - 出力上の `AnalysisFlowSteps` は helper から得た実行 step で作る。
5. `AnalysisRequestDispatcher` を新構造へ追従する。
   - 実行順は従来どおり simulation -> quality evaluation。
6. `ApplicationWorkflow` / `ApplicationTournamentUser` / `ProgramConsoleGuide` / `ManualAnalysisRequestReader` を追従する。
7. `StepRequests` 残存検索で旧依存が消えたことを確認する。

## 変えないこと

- シミュレーション計算ロジック。
- 品質評価ロジック。
- CSV / Markdown の出力形式。
- STSAInput/4 / STSAInput/5 の外部ファイル形式。
- `SimulationStepRequest` / `QualityEvaluationStepRequest` / `DeferredQualityEvaluationStepRequest` の具体入力内容。
