# プログラム4パート構成 第66ステップ実行計画

## 目的

第65ステップで追加した `AnalysisRequest` 直通ルートを、標準品質評価へ広げる。

これまで `STSAInput/4` の品質評価要求は、要求ファイルを legacy 入力列へ変換し、`InputFromSomewhere.UseText(...)` で対話入力を偽装していた。第66ステップでは、標準ルールの品質評価を `StandardQualityEvaluationRequest` として解析し、品質評価 mainline へ直接渡す。

## 対象範囲

今回の対象:

- `AnalysisFlowSteps=QualityEvaluation`
- `RuleProfileMode=Standard`
- `ExecutionMode=Single`
- `ExecutionMode=Sweep`
- 局数が20局を超える場合は `SimulationCount` が指定されている要求

今回の対象外:

- `RuleProfileMode=FinalStage` の品質評価
- `AnalysisFlowSteps=Simulation,QualityEvaluation` の複数ステップ要求
- `SimulationCount` なしで20局超の品質評価要求
- 手入力フローの要求オブジェクト化

対象外の要求は、従来どおり legacy converter へ fallback する。

## 実装方針

- `StandardQualityEvaluationRequest` を追加する。
- `StsaInput4RequestParser` で標準品質評価の `PlayersCsv` / `MatchesInput` / `ReferenceMatchesInput` / `Meta` / `Output` を解析する。
- 標準品質評価用の `TournamentQualityEvaluationRuleDefinition`、`TournamentQualityEvaluationInput`、`TournamentQualityEvaluationExecutionOptions`、`TournamentQualityEvaluationOutputOptions` を parser 側で構築する。
- `AnalysisRequestDispatcher` に `StandardQualityEvaluationRequest` の分岐を追加する。
- `TournamentQualityEvaluationMainline` に、入力済みオブジェクトを受け取る overload を追加する。
- 品質評価の依頼ログ出力は、対話で出力先を読んだ時点ではなく、レポート出力処理の中へ寄せる。

## 完了条件

- 標準品質評価 Single の STSAInput/4 smoke が、legacy 入力列を使わずに実行できる。
- 標準品質評価 Sweep の STSAInput/4 smoke が、legacy 入力列を使わずに実行できる。
- 標準シミュレーション新ルートが壊れていない。
- FinalStage 品質評価は fallback で動き続ける。
- `dotnet build` と `git diff --check` が成功する。
