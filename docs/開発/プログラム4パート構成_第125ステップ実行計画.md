# プログラム4パート構成 第125ステップ実行計画

## 目的

`STSAInput/5` の writer を追加し、手入力から保存する要求ファイルを `RuleProfileAttributes` セクション付きで出せるようにする。

第124ステップで parser は `STSAInput/5` の属性を直接持ち回れるようになった。保存側が `STSAInput/4` の互換 `RuleProfileMode` へ戻したままだと、入力形式と保存形式の差が残るため、要求オブジェクトから属性形式の行を生成する入口を追加する。

## 実施内容

1. `StsaInput4RequestWriter` に `BuildAttributeLines(...)` を追加する。
   - 既存 `BuildLines(...)` は `STSAInput/4` writer として維持する。
   - 単一ステップでは `RuleProfileAttributes` セクションを出す。
   - 複数ステップでは `Step.*.RuleProfileAttributes` セクションを出す。
2. 手入力の要求ファイル保存は `BuildAttributeLines(...)` を使う。
3. `--smoke-request-writer-roundtrip` に `--format STSAInput/5` を追加する。
   - 既存呼び出しは `STSAInput/4` のまま動く。
   - `STSAInput/5` 生成時も parser -> writer -> parser の再解析を確認できる。
4. `STSAInput/5` 仕様の writer 制約を更新する。

## 変えないこと

- 既存 `STSAInput/4` writer の出力形式。
- 品質評価の依頼ログ出力形式。依頼ログは引き続き `STSAInput/4`。
- 計算ロジック、入力 CSV、出力 CSV / Markdown の内容。

## 実施結果

- `StsaInput4RequestWriter.BuildAttributeLines(...)` を追加し、`STSAInput/5` の `RuleProfileAttributes` / `Step.*.RuleProfileAttributes` を出力できるようにした。
- 手入力の要求ファイル保存は `STSAInput/5` writer を使う形へ変更した。
- writer round-trip smoke に `--format STSAInput/5` を追加した。
- `STSAInput/5` 仕様の現在の制約を更新した。

- `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj`

- `dotnet run --project .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj -- --request-file "Output\SmokeGenerated\WriterStsa5\analysis_input_[先手8x後手8]_[Simulation_QualityEvaluation_STSAInput4_smoke].request.txt"`
