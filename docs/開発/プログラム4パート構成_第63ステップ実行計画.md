# プログラム4パート構成 第63ステップ実行計画

## 目的

`ApplicationWorkflow` に仮置きしている、現在のフレームワークと将来のフレームワークのスケルトンを、それぞれ責務に合う場所へ移し替える。

2026-06-01 時点では、`ApplicationWorkflow.Run()` の中に次の2系統が同居している。

- 現在の実装ルート: `ApplicationStartup.Start()` から `RequestWorkflow.Run(args)` へ進むルート
- 将来の実装スケルトン: `RequestBoundary` を作り、`RequestFileCheck` / `ManualInput` / `RequestFileCreate` / `AnalysisWorkflowNewVersion` へ進む4パート構成のルート

この同居は、設計を見比べるための一時状態として扱う。次ステップでは、`ApplicationWorkflow` を最上位の薄い入口に戻し、具体的な実行ルートは別クラスへ移す。

## 対象

- `ShogiTournamentSystemAnalyzer/Application/ApplicationWorkflow.cs`
- `ShogiTournamentSystemAnalyzer/Application/RequestWorkflow.cs`
- `ShogiTournamentSystemAnalyzer/Application/RequestSessionWorkflow.cs`
- `ShogiTournamentSystemAnalyzer/Application/Analysis/AnalysisWorkflowNewVersion.cs`
- 必要なら新規作成する実行ルート用クラス

## 移し替え方針

### 1. 現在の実装ルートを退避する

現行互換の実行ルートは、すぐに消さず、名前を付けて残す。

候補:

- `CurrentApplicationWorkflow`
- `LegacyApplicationWorkflow`
- `CompatibilityApplicationWorkflow`

中身は、現在 `ApplicationWorkflow` の上側にある次の処理を持つ。

```csharp
ApplicationStartup.Start();
RequestWorkflow.Run(args);
```

このルートは、既存の入出力とスモーク確認を壊さないための退避先にする。

### 2. 将来の4パート構成スケルトンを別クラスへ移す

`RequestBoundary` を作って、次の順に接続するスケルトンは、将来ルートとして分離する。

1. `RequestFileCheck`
2. `ManualInput`
3. `RequestFileCreate`
4. `Analysis`

候補:

- `FourPartApplicationWorkflow`
- `NextApplicationWorkflow`
- `ApplicationLifecycleWorkflow`

このクラスは、`docs/企画/【むずでょ原文】ライフサイクル案.md` の導線をコード上で表す場所にする。

### 3. `ApplicationWorkflow` は選択だけにする

`ApplicationWorkflow.Run(args)` は、実行ルートを選ぶだけの薄い入口に戻す。

短期的には、既存互換を優先して現行ルートを呼ぶ。

```csharp
CurrentApplicationWorkflow.Run(args);
```

将来ルートへ切り替える判断は、この第63ステップでは行わない。まずは同居状態を解消し、比較できる形にする。

## 注意点

- `bool isFutureImplementation` の意味とコメントが読み違えやすいので、移し替え時に整理する。
- `ApplicationWorkflow` から長い図コメントを直接持ち出し、将来ルート側か設計ドキュメント側へ寄せる。
- 既存挙動を変えない。最初の移動では、実行されるルートは現行互換のままにする。
- `AnalysisWorkflowNewVersion` は、将来ルートから呼ばれる候補として残す。現行ルートへ無理に接続しない。
- `RequestWorkflow` / `RequestSessionWorkflow` が現行互換のどの範囲を担当しているかを確認してから、名前を決める。

## 作業順

1. `ApplicationWorkflow` 内の2系統を読み分ける。
2. 現行互換ルートの退避先クラス名を決める。
3. 将来4パート構成ルートの退避先クラス名を決める。
4. 現行互換ルートを退避先へ移す。
5. 将来4パート構成スケルトンを退避先へ移す。
6. `ApplicationWorkflow` を、どちらかのルートを呼ぶだけの薄い入口へ戻す。
7. ビルドと既存スモークで、現行互換ルートが変わっていないことを確認する。

## 確認

- `dotnet build .\ShogiTournamentSystemAnalyzer\ShogiTournamentSystemAnalyzer.csproj`
- 空ルールの要求ファイル入力スモーク
- 手動入力保存のスモーク
- 既存の `RequestWorkflow.Run(args)` ルートが今まで通り動くこと

## 第63ステップで実装しないこと

- 将来4パート構成を本番ルートへ切り替えること
- `RequestWorkflow` / `RequestSessionWorkflow` の中身を大きく作り替えること
- `AnalysisWorkflowNewVersion` の本格接続
- 入力ファイル仕様の変更
- 出力仕様の変更

## メモ

このステップの主目的は、`ApplicationWorkflow` に置いてある比較用コードを片付けること。

現行ルートは壊さず残す。将来ルートはスケルトンとして読める場所へ移す。最終的に `ApplicationWorkflow` には、アプリケーション全体の入口として必要な最小限の分岐だけを残す。
