# プログラム4パート構成 第3ステップ実行計画

## 目的

第2ステップで `Analysis` を `Program.cs` から分離した。
第3ステップでは、`RequestFileCheck` を「要求ファイルを検査して、分析へ渡せる入力へ変換する入口」として明確にする。

## 現状

`InputSourceConfiguration` は `RequestFileCheck.TryApply(inputFilePath, ApplyInputFile)` を呼んでいる。
この名前と構造だと、`RequestFileCheck` 側が検査だけでなく `Console.SetIn()` まで含む入力適用をしているように見える。

実際には `Console.SetIn()` は `InputSourceConfiguration.ApplyInputFile()` の中で行われている。
ただし、`TryApply` という名前のため責務が読みにくい。

## 今回やること

1. 要求ファイルチェック結果を表す `RequestFileCheckResult` を追加する。
2. `RequestFileCheck.TryApply(...)` を `RequestFileCheck.TryRead(...)` に変える。
3. `TryRead` は、要求ファイルを読み、legacy 入力文字列へ変換できるかだけを確認する。
4. `Console.SetIn()` は `InputSourceConfiguration` の `ApplyCheckedInputFile(...)` に集約する。
5. `InputSourceConfiguration` の `ApplyInputFile` は `ReadInputFile` へ改名し、副作用をなくす。

## 今回やらないこと

- STSAInput の変換ロジック分離
- `InputSourceConfiguration` から巨大な変換関数群を移す作業
- 要求ファイルの警告や詳細診断の追加
- 入力ファイル失敗時のフォールバック方針変更

## 完了条件

- `RequestFileCheck` は検査結果を返す。
- `Console.SetIn()` は `InputSourceConfiguration` 側でだけ行う。
- `--input-file` 経由の smoke 実行が成功する。
- `dotnet build` が成功する。
