# プログラム4パート構成 第123ステップ実行計画

## 目的

`StsaInput4RequestParser` の `RuleProfileMode` 分岐を、互換ラベル読み取り後すぐ `RuleProfileAttributes` へ寄せる。

`RuleProfileMode` は `STSAInput/4` の互換入力として読み続けるが、要求ステップの種類を選ぶ判定では属性を使う形へ薄くする。

## 実施内容

1. 単一ステップ要求で、読み取った `RuleProfileMode` を `RuleProfileAttributes.FromCompatibilityLabel(...)` へ変換してから `ParseStepRequest` へ渡す。
2. 複数ステップ要求でも、各 `Step.*` セクションの互換ラベルを属性へ変換してから `ParseStepRequest` へ渡す。
3. `ParseStepRequest` の引数を `RuleProfileAttributes` に変更する。
4. Standard / FinalStage / TournamentFramework / Empty の実行先判定を、`SimulationShape` / `PairingSource` / `UsesFinalStageGrouping` で行う。
5. `STSAInput/5` の属性セクションから互換フォーマットへ入る既存ルートが引き続き動くことを smoke で確認する。

## 変えないこと

- `STSAInput/4` の `RuleProfileMode` 入力互換。
- `STSAInput/5` の属性セクション仕様。
- 標準 / 本戦 / 複数ステップの計算ロジック。
- 出力 CSV / Markdown の内容。

## 実施結果

- `StsaInput4RequestParser.ParseStepRequest(...)` は `RuleProfileMode` ではなく `RuleProfileAttributes` を受け取る形へ変更した。
- 互換ラベルは parser 入り口で読み取り続けるが、ステップ種別の判定は `IsStandardScheduledProfile(...)` / `IsFinalStageScheduledProfile(...)` と属性の `SimulationShape` へ移した。
- smoke 実行で生成された `Output` 配下の CSV / Markdown / `.stsa.txt` は、実装差分から外すため後始末した。
